# Soft Roadmap Guide — Post Current Roadmap

## Purpose
This document is a lightweight guide for direction after the current active roadmap is completed.

It is intentionally softer than the main roadmap.
It does not override active priorities.
It exists to preserve strategic direction without creating scope drift.

## Current Rule
Finish the current roadmap first.
Use this guide only for sequencing what comes next.

---

# Core Principle
Nova should continue optimizing one central mission:

**Useful AI with bounded authority, visible trust, and real daily value.**

If a future idea does not strengthen that mission, it should be deprioritized.

---

# Likely Next Phases After Current Roadmap

## 1. Trust UX Completion
Convert governance internals into visible user experience.

Examples:
- Request understanding card
- Action preview / consequence preview
- Better trust history surfaces
- Clear reasons for allow / deny / confirm
- Human-readable ledger views

Success looks like:
- User can see what Nova understood
- User can see what action would happen
- User can review what happened after execution

## 2. Frictionless Onboarding
Make first use dramatically easier.

Examples:
- One-command setup
- Guided installer
- First-run checks
- Model setup assistant
- Better recovery from setup failures

Success looks like:
- Install path is clear
- First useful result happens quickly
- Common failures are easy to recover from

## 3. Simplification Pass
Reduce unnecessary complexity.

Examples:
- Refactor hot files
- Remove duplicate docs
- Archive stale plans
- Improve naming consistency
- Reduce user confusion

Success looks like:
- Fewer confusing surfaces
- Easier maintenance
- Faster onboarding comprehension

## 4. One Killer Workflow
Choose one repeatable use case Nova wins at.

Examples:
- Trusted local executive assistant
- Safe desktop copilot
- Local family assistant
- Transparent operator console
- Research + action assistant

Selection criteria:
- Frequent need
- Easy to demo
- Better because of governance
- Valuable quickly
- Real repeat usage potential

## 5. Capability Maturity Program
Turn capability sprawl into clear readiness states.

Examples:
- Stable
- Beta
- Experimental
- Disabled by default
- Deprecated

## 6. Carefully Chosen Expansion
Only after trust + usability improve.

Examples:
- Better connectors
- Scheduling improvements
- Richer memory workflows
- Additional local tools
- Optional advanced operators

---

# Not Current Priority
- Broad connector expansion
- Enterprise packaging
- Mass consumer launch
- Experimental autonomy
- Chasing every new AI trend

---

# Guardrails

## Do Not Do
- Chase every new feature trend
- Add integrations without clear value
- Increase authority without visible controls
- Let docs overstate readiness
- Expand faster than maintainability allows

## Prefer
- Small finished wins
- Visible trust
- Strong defaults
- Reversible actions
- Honest product signals
- Tight scope discipline

---

# Decision Filter For Future Work
Before starting major work, ask:

1. Does this improve daily usefulness?
2. Does this improve visible trust?
3. Does this reduce friction?
4. Does this simplify the system?
5. Can this be finished well?

If most answers are no, reconsider the work.
