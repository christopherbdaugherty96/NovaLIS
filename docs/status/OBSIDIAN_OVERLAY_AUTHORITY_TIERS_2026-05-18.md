Status: proposed docs/tooling guidance / not runtime authority
Scope: Obsidian overlay navigation only

## Current truth

Nova currently treats the repository root as the Obsidian vault.

The Obsidian system is a generated overlay:

```text
scripts/generate_obsidian_overlay.py
_MOCs/
.obsidian/ local config generated by script and not treated as source truth
```

The overlay is useful because it lets docs and code appear in one graph while preserving the core boundary:

```text
Obsidian overlay = navigation and intelligence surface
Repo files = source materials
Generated runtime docs = highest authority for exact runtime state
Current continuity docs = highest authority for active work priority
```

Source files must not be edited by the overlay generator. Generated MOCs must remain rebuildable from repo files.

## Existing implementation strengths

The existing overlay already has important strengths:

- The repository root opens directly as the Obsidian vault.
- MOCs are generated rather than hand-maintained.
- Source docs and code are not modified by the generator.
- `.obsidian/` is local/generated config, not authoritative repo truth.
- Docs and code are connected in one graph.
- Navigation exists for folders, categories, phases, topics, code layers, imports, and tests.
- Contract tests cover generator behavior.
- Tests protect generated links and source-note immutability.

These strengths should be preserved. The next patch should add authority-tier navigation, not replace the current overlay model.

## Problem

The current overlay is good at grouping by folder, category, phase, topic, code layer, imports, and tests.

The missing layer is authority rank.

Without authority rank, Obsidian can make all notes look equally valid:

```text
current runtime truth
active priority docs
proof packets
future planning docs
archived docs
historical branch notes
```

That is risky for Nova because future or historical documents can visually resemble current truth.

## Required principle

Obsidian must never become runtime authority.

It may:

- help navigation
- surface links
- group related files
- show code/doc relationships
- support review and research
- make current truth easier to find

It may not:

- override generated runtime docs
- promote future docs into current truth
- imply a capability is certified or locked unless lock docs say so
- authorize runtime behavior
- authorize OpenClaw expansion
- authorize browser/computer-use expansion
- authorize external writes
- authorize autonomous execution

## Authority tier model

### TIER 0 — Generated runtime truth

Highest runtime-truth layer.

Examples:

```text
docs/current_runtime/CURRENT_RUNTIME_STATE.md
docs/current_runtime/RUNTIME_FINGERPRINT.md
docs/current_runtime/GOVERNANCE_MATRIX.md
docs/current_runtime/BYPASS_SURFACES.md
docs/current_runtime/SKILL_SURFACE_MAP.md
```

Rules:

- Generated runtime docs outrank human-written status docs for exact runtime state.
- Manual edits to generated runtime truth are not allowed.
- Exact capability counts, hashes, surfaces, and execution-path facts should be delegated to generated runtime docs.

### TIER 1 — Current operational continuity

Current working-state layer.

Examples:

```text
AGENTS.md
.agent_context/current_priority.md
docs/status/CURRENT_WORK_STATUS.md
docs/todo/ACTIVE_TODO.md
```

Rules:

- These docs identify the current priority, current branch/lane, next action, and continuity state.
- They must avoid copying exact generated runtime counts or hashes unless intentionally linking to generated truth.
- They are current-work truth, not runtime execution authority.
- If TIER 1 and TIER 0 appear to disagree about runtime facts, TIER 0 wins for runtime facts and TIER 1 should be patched.

### TIER 2 — Accepted evidence, locks, and proof records

Accepted proof and review layer.

Examples:

```text
docs/PROOFS/**
docs/capability_verification/**
docs/status/*LOCK*.md
docs/status/*CLOSEOUT*.md
docs/audits/*.md
```

Rules:

- Proof records can support claims but do not override generated runtime truth.
- Audit docs remain evidence records unless explicitly promoted by current continuity docs.
- Capability status must distinguish active, certified, and locked.
- Audit findings should not be treated as fixed unless a later implementation, proof, or closeout record confirms the fix.

### TIER 3 — Future / planning only

Planning and direction layer.

Examples:

```text
docs/future/**
docs/design/** when describing future design rather than current implementation
roadmaps not referenced by current continuity docs
```

Rules:

- Future docs are not current runtime authority.
- They must not be interpreted as implemented unless code, generated runtime truth, and current status docs confirm implementation.
- They must not authorize new writes, autonomy, connectors, browser/computer-use, OpenClaw expansion, or capability expansion.

### TIER 4 — Historical / archive

Historical context layer.

Examples:

```text
docs/archive/**
archive/**
stale branch notes
superseded roadmaps
old pass reports not referenced by current continuity docs
```

Rules:

- Historical docs are useful for context but not for current truth.
- They should not drive current work unless revived through a new reviewed priority lock.
- Obsidian should visually or structurally separate these from active/current docs.

## Recommended generated overlay additions

The Obsidian generator should eventually add a generated file:

```text
_MOCs/AUTHORITY_TIERS.md
```

That generated MOC should group files by authority tier and link back to:

```text
_MOCs/HOME.md
_MOCs/USER_PATHS.md
_MOCs/BY_TYPE.md
_MOCs/REPO_BY_FOLDER.md
```

`_MOCs/HOME.md` should include an early link such as:

```text
- [[_MOCs/AUTHORITY_TIERS|Read by authority tier — current truth vs planning vs archive]]
```

This should appear before broad browsing links so new readers start from truth rank before graph exploration.

## Classification gaps to fix later

Current generated categories are useful but not enough.

Known gaps:

```text
Reference is too broad.
docs/future should be first-class planning-only.
Phase views can mix shipped and planned docs.
Audit docs can look current even when historical.
Archive/history needs stronger visual separation.
```

Recommended future generator classifications:

```text
current_truth
current_status
proof_or_lock
future_planning
historical_archive
reference
```

These should supplement, not replace, existing topic/category/code-layer views.

## Branch handling conclusion

Do not merge stale Claude/Kodak Obsidian or status branches directly merely because they refreshed MOCs.

Before merging any Obsidian/status branch:

```text
1. Compare against current main.
2. Check whether later PRs already superseded the work.
3. Keep only non-duplicated guidance or generator changes.
4. Do not merge stale generated MOCs over newer generated output.
5. Do not merge old future/planning wording without current-truth review.
```
