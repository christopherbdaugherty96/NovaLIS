# Nova Cross-System Memory and Governed Awareness Direction
Date: 2026-03-27
Status: Cross-phase design direction
Scope: How Nova evolves from stateful-in-a-repo to stateful-across-the-environment without breaking governance

## 1. Purpose
This document captures the missing product step between:
- a useful coding or project assistant that is stateful inside one repo
- and a true Nova system that remains stateful, aware, and helpful across the user's broader environment

The goal is to define that step clearly enough that it can guide future implementation without blurring memory, autonomy, and authority.

## 2. Design Status Boundary
This is a design-direction packet, not runtime proof.

Read it this way:
- current runtime truth still lives in `docs/current_runtime/CURRENT_RUNTIME_STATE.md`
- current memory and continuity runtime work is grounded in Phase 5
- current trust-loop and review visibility runtime work is grounded in Phase 6
- current bounded assistive noticing and advanced Governor planning are grounded in Phase 8 and later

This packet exists to connect those tracks into one understandable product direction.

## 3. The Missing Piece
The missing product step is:

> Cross-system memory plus governed awareness

That means Nova should eventually:
- remember work across sessions
- understand what the user is doing over time
- notice patterns and repeated friction
- help when appropriate
- but never act without permission

The important difference is that this is bigger than "AI with memory."
It is a governed continuity system.

## 4. The Real Evolution
The clean model is:

### Stage 1 - Stateful inside one repo or project
The assistant remembers:
- repo context
- docs
- recent coding steps
- local workflow within one project boundary

This is where many coding-agent products stop.

### Stage 2 - Stateful across the user's environment
Nova remembers and interprets:
- current work across sessions
- continuity between project threads
- durable preferences and rules
- recurring blockers and friction patterns
- runtime trust conditions

This is the real Nova direction.

### Stage 3 - Stateful across the physical world
This is the later robotics / embodied-systems direction:
- device state
- trusted physical environments
- physical routines
- real-world sensor and actuator boundaries

That stage belongs later and should only be built on top of the same Governor-first model.

## 5. Why This Feels Bigger
This is no longer just:

> How do I get better answers from AI?

It becomes:

> How do I live and work with AI safely, consistently, visibly, and under my control?

That is a different category of system.

It is closer to:
- a continuity layer
- a trust layer
- an authority layer
- a bounded personal operating layer

## 6. What To Take From Stateful AI Systems
There are valuable ideas to reuse from stateful coding-assistant systems:
- structured memory
- reusable workflows
- strong documentation
- visible guardrails
- self-improvement in the narrow sense of better workflow adaptation under review

Applied to Nova, those ideas become:
- structured memory -> governed memory and remembrance layers
- reusable workflows -> governed capabilities and bounded templates
- guardrails -> Governor, ledger, permissions, and review surfaces
- documentation -> canonical design and runtime truth locks

## 7. What Not To Take
Nova should not import the parts that would break its philosophy:
- silent self-training
- implicit learning
- hidden preference accumulation
- uncontrolled behavior drift
- silent adaptation that changes authority boundaries

Nova should not behave like:
- a system that quietly rewires itself
- a system that acts because it inferred intent
- a system that accumulates hidden user models without permission

## 8. The Correct Upgrade Path For Nova
The right path is:

### 8.1 Memory with permission
Nova should support:
- explicit save
- explicit remember
- optional save suggestions
- export and delete

But not:
- silent learning
- silent long-term profiling

### 8.2 Operational remembrance
Nova should visibly track:
- what it is doing
- what failed
- what is ongoing
- what the current continuity anchor is

This is remembrance for runtime coherence, not personhood simulation.

### 8.3 Assistive noticing
Nova should detect:
- repeated friction
- repeated blockers
- repeated missing setup steps
- repeated continuity gaps
- trust conditions

But it should only:
- surface
- explain
- suggest
- ask

Never silently act.

### 8.4 Skill awareness, but governed
Nova should become more aware of what it can do.

But this should be expressed as:
- governed capabilities
- registry-controlled access
- execution-bound tooling
- policy-visible surfaces

Not as:
- unconstrained free-form skill drift

## 9. Cross-Phase Ownership
This direction spans multiple phases and should stay separated accordingly.

### Phase 5
Owns:
- explicit memory
- durable user-approved remembrance
- project continuity
- working context

### Phase 6
Owns:
- trust-visible continuity
- review surfaces
- policy-visible runtime state
- auditable operational remembrance

### Phase 8
Owns:
- bounded assistive noticing
- advanced Governor awareness
- trust scoring / anomaly-aware later direction
- help surfaces that remain bounded and non-autonomous

### Phase 9 and later
Own:
- richer cross-system connectors
- broader real-world context
- physical-world and robotics expansion

## 10. Governing Rule
The clean governing rule is:

> Nova may remember with permission, notice with discipline, and help with restraint.

That implies:
- memory is explicit or explicitly approved
- awareness is visible and auditable
- suggestions are bounded
- authority never comes from inference alone

## 11. Product Truth
Nova is not trying to become "just a smarter chatbot."

Nova is trying to become:
- a stateful continuity layer
- a governed memory layer
- a trust-preserving awareness layer
- a safe operator layer over intelligence

That is why the system feels different from tools that focus only on better answers.

## 12. Implementation Direction
The practical implementation order remains:
1. explicit memory and exportability
2. operational remembrance visibility
3. bounded assistive noticing
4. handled-state controls and calm surfacing
5. richer capability awareness
6. broader cross-system connectors
7. later physical-world expansion

This ordering matters because visibility and restraint have to arrive before broader autonomy pressure does.

## 13. Final Statement
Most systems optimize for:

> better answers

Nova is optimizing for:

> when answers are allowed, remembered, trusted, surfaced, and acted on

That is the right foundation for a local-first, governed, continuous AI system.
