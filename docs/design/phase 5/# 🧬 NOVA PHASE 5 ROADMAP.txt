## 🔹 Executive Summary  

Phase 5 introduces **user‑sovereign memory and continuity** – the first time Nova can retain information across sessions, adapt its tone per domain, and offer structured continuity, **all under explicit user control**.  

This phase is split into two parallel tracks to ensure experience maturity keeps pace with memory capability:  

- **Phase 5A – Memory & Continuity:** Explicit memory (save/lock/delete), declarative identity, pattern detection (quarantine → explicit approval), per‑domain tone calibration.  
- **Phase 5B – Experience Maturation:** Notification design, ambient presence, multi‑modal integration, household rhythm, recovery narratives.  

All memory and adaptation remain **explicitly user‑approved** – no inference, no behavioral learning, no background routines. Nova’s autonomy remains at **Tier 2 (Observational)** at most, with mutations limited to **Class C (UI Formatting)** and manual approval. The Governor remains the sole authority for all memory operations and tone mutations.  

Phase 5 is a **design‑only phase** – its components may be implemented in code but remain **runtime‑locked** until Phase 4.5 is complete and stable, and the Phase 5 admission gate is satisfied. Runtime unlock will occur in a future phase (likely Phase 6) after proven household value.  

---

## 🎯 Phase 5 Goals  

1. **Establish explicit, inspectable, revocable memory** – users can save, lock, defer, and delete information.  
2. **Enable per‑domain tone calibration** – Nova’s response style adapts to context (calendar, technical, casual) through explicit user feedback and quarantine‑then‑approve.  
3. **Provide declarative identity configuration** – users can set preferred name, voice, and communication style.  
4. **Introduce pattern detection without auto‑apply** – Nova may detect statistical patterns from explicit interactions and present them in a quarantine queue for user review.  
5. **Design calm, opt‑in notification and ambient presence** – users can choose to receive gentle alerts (never proactive).  
6. **Develop multi‑modal coherence** – voice, screen, and ambient display work together, with offline‑first preservation.  
7. **Build recovery narratives** – explain failures in plain language, maintaining trust.  
8. **Maintain all Tier 1 invariants** – no autonomy, no background cognition, no predictive preloading, no silent fallback, no authority expansion.

## 🏛️ Architectural Overview  

Phase 5 extends the Phase 4 and Phase 4.2 architecture with new memory subsystems and tone profiles, while preserving the Governor as the single choke point.  

```
User Input → GovernorMediator
    → (if memory operation) → MemoryGovernor (sub‑component)
    → (if tone feedback) → ToneProfileManager
    → (else) → Personality Agent / Agent Stack
```

### Memory Governance Layer  

- **MemoryStore:** Filesystem‑based storage (or encrypted local DB) with three directories: `locked/`, `active/`, `deferred/`.  
  - **Hardening:** MemoryStore rejects writes if GovernorMediator signature not present.  
- **MemoryGovernor:** Mediates all memory operations; enforces access rules (read only on explicit request; write only on user command).  
- **MemoryAgent** (Phase 4.2) becomes the cognitive interface to memory – retrieves items only when user asks, never proactively.  

### Tone Calibration Architecture  

- **ToneProfile** per domain (Ring 2, `@mutable`, Class C). Stores offsets: `verbosity_offset`, `explanatory_depth_offset`, `formality_offset`.  
- **ToneProfileManager:** Applies domain‑specific profile to Personality Agent output after domain detection (via skill routing).  
  - **Deterministic application:** Transformations are rule‑based, not adaptive.  
- **PatternDetector** (Tier 2): Observes **explicitly stored memory and explicit interaction logs** (opt‑in only), detects statistical patterns (e.g., “user frequently expands weather details”), writes to quarantine queue.  
  - **Prohibited:** Analyzing transient session content unless user has enabled pattern detection; inferring personality, intent, or emotional traits.  
- **QuarantineQueue:** User‑accessible list of detected patterns; user can approve (→ updates domain profile via Class C mutation) or reject (→ discarded).  

### Notification & Ambient Presence  

- **NotificationService:** Manages opt‑in user‑defined notifications (e.g., “remind me at 8am”).  
  - **Scheduling constraint:** Relies on externally triggered system timers (e.g., cron, launchd) or OS‑level scheduled tasks; Nova does not run persistent background loops. Governor validates and registers the timer, but the timer itself is external.  
- **AmbientUI:** Subtle UI elements (e.g., Trust Panel border color change) that reflect system state without demanding attention.  

### Multi‑Modal Integration  

- **VoiceRenderer:** Applies surface‑specific **deterministic** transformations (short sentences, no bullets) based on final tone profile.  
- **ScreenRenderer:** Uses same tone profile but can display tables, bullet lists.

## ⚠️ Risks and Mitigations  

| Risk | Mitigation |
|------|------------|
| **Memory inference** | MemoryAgent only returns items on explicit user request; no automatic context retrieval. Governor signature required for writes. |
| **Tone calibration drifts into persuasion** | All mutations are Class C (formatting only); validation pipeline blocks authority language; deterministic transformation only. |
| **Pattern detection becomes behavioral profiling** | Opt‑in only; operates only on explicit interactions; may not infer intent/emotion; quarantine ensures human review. |
| **Notification spam** | Opt‑in only, user‑defined; Governor enforces rate limits; scheduling uses external timers, not Nova loops. |
| **Ambient cues misinterpreted as semantic** | All ambient changes pass Four Gates; tested with users for perceived meaning. |
| **Multi‑modal inconsistency** | Shared tone profile ensures coherence; surface adaptations are fixed rules, not learned. |
| **Background scheduling leaks autonomy** | External timers only; Nova never wakes itself; Governor registers but does not execute. |

## ✅ Success Criteria for Phase 5 Design Completion  

- All memory operations implemented and tested; Ledger records every operation; Governor signature required for writes.  
- Per‑domain tone profiles functional; pattern detection opt‑in, quarantine queue operational; approve/reject flow works.  
- Declarative identity configurable and applied.  
- Notifications and ambient cues pass Four Gates; user studies confirm calm perception; notifications use external timers only.  
- Multi‑modal integration preserves offline‑first and explicit invocation; surface adaptations deterministic.  
- Recovery narratives tested for calmness and clarity.  
- Constitutional audit confirms no new authority, no autonomy, no inference, no background loops.  
- All components runtime‑locked (inactive until Phase 6 unlock).  

---

## 🔮 After Phase 5: Path to Phase 6  

Phase 5 components will remain dormant until **Phase 6 (Policy‑Bound Operation)** unlocks runtime for memory, tone calibration, and notifications. Phase 6 will introduce:  

- **Delegated autonomy** within explicit user‑defined policies.  
- **Bounded automation** (e.g., “every morning at 8am, summarise my calendar”).  
- **Local optimisation** (Class A/B/D mutations with evolution ON).  

Phase 6 admission will require Phase 4.5 and Phase 5 completion, plus ≥3 months of stable household deployment.
