# NOVA PHASE 5 ROADMAP (Corrected)
## Memory and Continuity - User-Sovereign Persistence

Document Status: Proposed - Design Not Yet Authorized
Prepared by: Nova Constitutional Governance Board
Revised: 2026-03-11

## Executive Summary

Phase 5 introduces user-sovereign persistence and continuity under strict governance. It adds explicit memory operations, inspectable adaptation controls, and continuity tooling while preserving constitutional invariants:

- no authority expansion
- no hidden persistence
- no background autonomy
- no execution outside Governor mediation

This roadmap remains proposed. No Phase-5 persistent capability is runtime-authorized until the admission gate is ratified.

## Track Structure

Phase 5 is split into two tracks:

- Phase 5A - Memory and Continuity:
  - explicit save/lock/defer/delete
  - memory governance tiers and schemas
  - user-inspectable continuity controls
  - declarative identity preferences
- Phase 5B - Experience Maturation:
  - calm notification design (opt-in and user-scheduled)
  - multi-surface coherence
  - trust-facing explainability and recovery narratives

## Phase Boundary Clarification

Working Context Engine is Phase-4.5-compatible runtime behavior and is session-scoped only.

- Working Context Engine: non-persistent, session-local task coherence
- Phase 5 Memory: cross-session persistence with explicit user governance

This distinction is non-negotiable: Phase-4.5 context coherence must not silently become Phase-5 persistent memory.

## Goals

1. Establish explicit, inspectable, revocable memory.
2. Keep memory as filing, not learning.
3. Enable bounded tone calibration with transparent controls.
4. Support continuity across user-approved project threads.
5. Preserve all constitutional Tier-1 invariants.

## In Scope (Proposed)

- Memory tiers: locked, active, deferred
- Memory operations with confirmation rules
- Continuity views and project thread status surfaces
- Tone profile controls with inspectable mutation logs
- Notification scheduling model that is user-invoked or user-scheduled only

## Out of Scope (Proposed)

- autonomous/background execution
- predictive preloading without invocation
- implicit delegation
- learning-based authority expansion
- hidden or non-revocable adaptation

## Architectural Overview

Authority plane remains unchanged:

`User -> brain_server -> GovernorMediator -> Governor -> ExecuteBoundary -> Executor -> Ledger`

Phase-5 additions are governance-constrained data layers:

- Memory plane (persistent, explicit, inspectable)
- Continuity plane (thread/project state reasoning)
- Tone calibration plane (Class C formatting mutations only)

All mutations remain reversible and ledger-visible.

## Data Boundary Rules

Memory and tone adaptation must remain isolated by contract:

- Persistent memory contents must not directly drive tone shifts.
- Tone calibration inputs are limited to explicit feedback and approved calibration signals.
- Tone adaptation must not change authority, policy, or capability routing.

## Admission Gate (Required Before Design Unlock)

- [x] Phase 4 runtime unlocked and stable for >=30 days.
- [x] Phase 4.5 design completed and ratified.
- [x] Phase 4.5 user research confirms daily-use scenarios and calm perception.
- [ ] Memory Governance specification ratified.
- [ ] Tone Calibration architecture approved.
- [ ] Pattern detection opt-in and constraints ratified.
- [ ] Notification scheduling boundary ratified.
- [ ] Constitutional audit confirms no authority expansion and no hidden autonomy.

Design Unlock Act: Phase-5 implementation design can begin only after all gate checks are complete and board ratification is issued.

## Implementation Policy

- Phase-5 code paths must be behind compile-time or runtime phase gates.
- Persistent memory and adaptive tone mutations remain runtime-locked until explicit unlock.
- Exception: Working Context Engine remains an existing Phase-4.5 session-scoped runtime component and does not constitute Phase-5 persistence.

## Risks and Mitigations

- Risk: scope creep from context into persistence.
  - Mitigation: explicit boundary tests for session-only context modules.
- Risk: silent adaptation drift.
  - Mitigation: ledger events + inspectable settings + user override.
- Risk: notification behavior interpreted as proactive autonomy.
  - Mitigation: user-invoked/user-scheduled delivery only; no autonomous push loops.

## Success Criteria for Phase-5 Design Completion

- Memory governance contract finalized and testable.
- Tone calibration boundaries and visibility model finalized.
- Continuity model remains user-sovereign and revocable.
- No constitutional invariant regressions.
- Proof packet prepared for unlock decision.

## Path to Phase 6

Phase-5 design outputs remain governed and phase-gated. Runtime unlock for persistent memory/adaptation is deferred to a later policy-bound phase after additional safety proofs.
