3. ARCHITECTURE OVERVIEW

```
[Sensors/Devices] ↔ [Zigbee2MQTT] ↔ [Mosquitto MQTT]
                                    ↙        ↓
                        [Home Assistant]   [Governor Core]
                                              ↓
                                ┌─────────────┴─────────────┐
                                ↓                           ↓
                       [Interface Layer]             [Service Supervisor]
                     Voice / Dashboard / LLM           (health checks)

```

· Core Layer (always running): Governor Core subscribes directly to zigbee2mqtt/# topics for device events. Home Assistant subscribes to the same topics for UI/automation but is not the source of truth for Nova.
· Interface Layer (restartable): Voice service, dashboard, and LLM run in separate containers. If any crash, the Core continues unaffected.
· Service Supervisor: A lightweight container that performs health checks on all services and restarts them if needed. Ensures appliance‑grade uptime.

4. PHASE 7 BUILD: STEP BY STEP

Phase 0 – Base System & Docker

1. Flash Raspberry Pi OS Lite (64‑bit) to SSD using Raspberry Pi Imager. Enable SSH, set hostname nova.local.
2. Boot Pi, update: sudo apt update && sudo apt upgrade -y
3. Install Docker and Docker Compose:
   ```
   curl -fsSL https://get.docker.com -o get-docker.sh
   sudo sh get-docker.sh
   sudo usermod -aG docker $USER
   ```
4. Log out and back in, then create a project directory: mkdir ~/nova && cd ~/nova

Phase 1 – Home Automation Backend (All Containers)

Create a docker-compose.yml with the following services:

```yaml
version: '3.8'

services:
  mosquitto:
    image: eclipse-mosquitto
    container_name: mosquitto
    restart: unless-stopped
    ports:
      - "1883:1883"
      - "9001:9001"
    volumes:
      - ./mosquitto/config:/mosquitto/config
      - ./mosquitto/data:/mosquitto/data
      - ./mosquitto/log:/mosquitto/log

  zigbee2mqtt:
    image: koenkk/zigbee2mqtt
    container_name: zigbee2mqtt
    restart: unless-stopped
    devices:
      - /dev/ttyACM0:/dev/ttyACM0
    volumes:
      - ./zigbee2mqtt/data:/app/data
      - /run/udev:/run/udev:ro
    environment:
      - TZ=America/New_York

  homeassistant:
    image: ghcr.io/home-assistant/home-assistant:stable
    container_name: homeassistant
    restart: unless-stopped
    network_mode: host
    volumes:
      - ./homeassistant/config:/config
      - /etc/localtime:/etc/localtime:ro
```

1. Start the services: docker-compose up -d
2. Configure Zigbee2MQTT web interface (port 8080) to pair devices. They will appear in Home Assistant automatically.
3. Add to Home Assistant configuration.yaml:
   ```yaml
   mqtt:
     broker: localhost
   ```

Phase 2 – Governor Core (Container)

Create a new service in docker-compose.yml:

```yaml
  governor:
    build: ./governor
    container_name: governor
    restart: unless-stopped
    ports:
      - "8000:8000"
    volumes:
      - ./governor/data:/data
      - ./governor/ledger:/ledger
    depends_on:
      - mosquitto
```

Create ./governor/Dockerfile:

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
CMD ["uvicorn", "governor:app", "--host", "0.0.0.0", "--port", "8000"]
```

requirements.txt:

```
fastapi
uvicorn
websockets
paho-mqtt
aiofiles
redis
```

Implement governor.py with:

· FastAPI app with WebSocket endpoint for dashboard
· MQTT client subscribing to zigbee2mqtt/# (not homeassistant/#)
· SQLite database for capability registry and ledger
· Governor mediation: parse user input, match capability, check permissions, log to ledger
· Health endpoint /health returning 200
· In‑memory state graph for recent events (optional Redis for persistence)

Phase 3 – Voice Interface (Container)

Add voice service to docker-compose.yml:

```yaml
  voice:
    build: ./voice
    container_name: voice
    restart: unless-stopped
    devices:
      - /dev/snd:/dev/snd
    volumes:
      - ./voice/models:/models
    depends_on:
      - governor
```

./voice/Dockerfile:

```dockerfile
FROM python:3.11-slim
RUN apt-get update && apt-get install -y portaudio19-dev
WORKDIR /app
COPY requirements-voice.txt .
RUN pip install --no-cache-dir -r requirements-voice.txt
COPY . .
CMD ["python", "voice_service.py"]
```

requirements-voice.txt:

```
pvporcupine
vosk
piper-tts
pyaudio
requests
```

Implement voice_service.py with:

· Porcupine wake word detection
· Record audio, send to Vosk
· Send transcribed text to Governor endpoint /command
· Receive response, speak with Piper
· Health endpoint /health

Phase 4 – Intelligence Layer (Optional Container)

Add Ollama service (optional):

```yaml
  ollama:
    image: ollama/ollama:latest
    container_name: ollama
    restart: unless-stopped
    volumes:
      - ./ollama/models:/root/.ollama
    ports:
      - "11434:11434"
```

Governor will call Ollama only when explicitly requested (e.g., "summarize today's events"). Wrap calls with timeout and fallback.

Rule: If Ollama is down, Nova continues operating normally. All core functions (voice, device control, dashboard) must work without LLM.

Phase 5 – Dashboard

The dashboard is served by Governor as static files. Create ./governor/static/index.html with:

· System health (via /api/system)
· Latest sensor events (via WebSocket)
· Proposal queue
· Trust Panel (active monitoring domains, online/offline status)

Governor will serve static files and push WebSocket updates.

Phase 6 – Continuous Awareness (Opt‑In)

In Governor, maintain an in‑memory state graph (e.g., Python dict with timestamped values). Subscribe to MQTT zigbee2mqtt/# for real‑time updates.

Implement simple anomaly detection:

· Temperature out of range
· Motion at unusual hours
· Disk filling faster than normal

Add detected anomalies to a proposal queue (visible in dashboard). Do not speak or alert automatically.

Add a Trust Panel toggle to enable/disable continuous awareness (default off). When enabled, show which domains are monitored (motion, temp, etc.).

Ensure all observation data is stored only ephemerally (Redis or in‑memory) unless explicitly approved for trend storage.

Phase 7 – Service Supervisor

Create a separate container (or systemd service) that performs health checks:

```yaml
  supervisor:
    image: alpine:latest
    container_name: supervisor
    restart: unless-stopped
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
    command: >
      sh -c "apk add curl && while true; do
        for service in governor voice ollama; do
          if ! curl -f http://$service:8000/health; then
            docker restart $service
          fi
        done
        sleep 30
      done"
```

This checks each service's health endpoint every 30 seconds and restarts if unhealthy.

Phase 8 – Log Rotation

Prevent disk fill by adding Docker log limits in docker-compose.yml globally or per service:

```yaml
x-logging: &default-logging
  options:
    max-size: "10m"
    max-file: "3"
  driver: json-file

services:
  mosquitto:
    logging: *default-logging
  ...
```

Also configure system logrotate for any non‑container logs.

5. WHAT PHASE 7 ACTUALLY FEELS LIKE

· Nova is always on – Pi consumes ~3‑5W, silent, no fan needed if properly cooled.
· Dashboard at http://nova.local:8000 shows current home state, recent events, calm insights.
· Voice: Say "Nova, what's the temperature?" → wake word → Vosk → Governor → (cached sensor) → Piper responds in 1‑2 seconds.
· Continuous awareness (opt‑in) enables Nova to notice patterns: "Front door opened at 2am (unusual)". This appears in the proposal queue, not as an alert.
· Actions require explicit confirmation: "Turn on kitchen light" → Governor checks capability → sends MQTT command to Zigbee2MQTT → light on. Dashboard updates instantly.
· Trust Panel always visible: shows "Continuous Monitoring: Motion, Doors, Temp" when enabled.
· LLM used sparingly – only for summaries or when asked "explain". It runs at 5‑10 tokens/sec on Pi 5 with 2‑3B models – usable but not chatty. If Ollama fails, Nova falls back gracefully: "I'm unable to generate a summary right now."

6. PERFORMANCE NOTES

· LLM: For smooth experience, use 2‑3B quantized models. Avoid 7B+ unless you have a separate machine.
· Voice: Porcupine + Vosk small model keeps CPU low (<10% when idle). Whisper is heavier; use only if you need higher accuracy.
· Home Assistant + MQTT: Very lightweight (<100MB RAM).
· Governor: Thin; should be fine with <50MB RAM.
· Total RAM usage (with small LLM not constantly loaded): ~1‑1.5GB, leaving plenty for system.

Recommendation: Run LLM on‑demand (load only when needed) to save RAM.
