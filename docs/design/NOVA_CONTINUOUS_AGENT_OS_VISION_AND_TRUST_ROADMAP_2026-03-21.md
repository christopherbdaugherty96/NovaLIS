# Nova Continuous Agent OS Vision And Trust Roadmap
Date: 2026-03-21
Status: Planning packet only; cross-phase vision for Nova as a continuously available personal agent system
Scope: Preserve the integrated vision of Nova as a governed work assistant, personal assistant, home-integrated system, and eventually adaptive agent without losing trust or explicit user control

## Purpose
This packet captures a larger product vision:

Nova is not only becoming an assistant.
Nova is becoming a continuously available personal agent system.

The target experience is something closer to:
- a continuously-on Codex-style work partner
- a conversational assistant
- a project-aware personal system
- a governed home-integrated assistant
- one stable identity with growing capability

This is not runtime truth.
It is the roadmap for how the existing Nova work should fit together into one trust-centered system.

## Core Vision
In plain English, the long-term goal is:

Nova should be able to:
- think and work with you
- read and update project files and documents
- help with GitHub and coding workflows
- connect to common apps and sources
- stay aware of your active work
- remember explicit things you asked it to remember
- use those memories when relevant
- help with local and home integrations
- maintain one recognizable personality
- eventually adapt and learn under reviewable controls

But it should do all of that while staying:
- governed
- explicit
- inspectable
- approval-aware
- user-controlled

## The Most Important Product Rule
The right way to build this is not:
- one giant blob of intelligence with unclear authority

The right way is:
- one Nova identity
- multiple separated systems underneath
- one trust model across all of them

That is how Nova can become more capable without becoming unsafe or chaotic.

## What Nova Is Really Becoming
Nova is evolving into a combination of:

### 1. A Codex-Style Work Agent
Nova should:
- read code
- update docs
- generate reports
- work with repos
- understand project structure
- help with GitHub workflows

### 2. A Conversational Assistant
Nova should:
- explain clearly
- research well
- follow context
- answer naturally
- handle follow-ups and clarifications well

### 3. A Home And Local Environment Assistant
Nova should:
- understand the local system
- perform governed local actions
- eventually integrate with home systems
- keep approvals explicit where real-world effects exist

### 4. A Personal Operating Layer
Nova should also become:
- project-aware
- memory-aware
- connector-aware
- schedule-aware
- trust-aware

This is what turns Nova into a real personal agent system rather than a collection of disconnected features.

## The Right Architecture
To do this safely, Nova needs layered systems.

### 1. Identity Layer
This is:
- Nova's voice
- personality stack
- tone
- stable identity

This layer should stay consistent.

### 2. Conversation Layer
This is:
- topic tracking
- clarification strategy
- answer planning
- session continuity
- conversation flow

This layer makes Nova feel natural and coherent.

### 3. Project / Work Layer
This is:
- repos
- documents
- reports
- active workspace state
- GitHub and project context
- task continuity

This layer makes Nova able to work the way a continuously-on coding and writing partner should work.

### 4. Memory Layer
This is:
- `save this`
- `remember this`
- memory viewing
- memory editing/deleting
- relevant recall
- explicit durable notes

This layer should begin as explicit governed memory, not hidden learning.

### 5. Connector Layer
This is access to:
- GitHub
- local folders/repos
- plain text files
- Word docs
- Google Docs
- Notion
- Obsidian
- VS Code / workspaces
- web/news/search sources
- calendars and other sources later
- home systems later

### 6. Action Layer
This is:
- local system actions
- file and app opens
- governed write/update actions
- home actions
- approval-gated real-world effects

This is where Nova's governor matters most.

### 7. Background / Always-On Layer
This is:
- schedules
- recurring tasks
- reminders
- project refreshes
- daily brief generation
- state checks
- notifications

This is the layer that makes Nova feel continuously available rather than only reactive.

### 8. Trust / Governor Layer
This is:
- permissions
- approvals
- connector boundaries
- risk levels
- audit logs
- visible trust controls

This is what keeps the whole system coherent and safe.

### 9. Learning / Adaptation Layer
This is much later:
- durable preference shaping
- adaptive retrieval
- personality refinement
- learning from long-term use

This should not arrive before explicit review, rollback, and audit controls.

## What "Continuously On" Should Mean
The right version of "continuously on" is not:
- uncontrolled background autonomy

The right version is:
- always available
- project-aware
- memory-aware
- task-aware
- able to continue ongoing work
- able to run user-approved recurring tasks
- still bounded by the same trust model

Nova should feel present and ready, not secretly self-directing.

## Best Product Recommendation

### Build Nova As A Personal Agent Operating System
The strongest framing for this vision is:

Nova should become a personal agent operating system.

That means:
- one assistant identity
- multiple governed subsystems
- one trust model
- one place to work with projects, memory, connectors, tasks, and actions

### Build The System Around Nova, Not Just More Features
The biggest future gains will not come from isolated new tricks.

They will come from:
- projects
- connectors
- memory
- schedules
- permissions
- evals
- context management
- mode separation

That is the real operating framework Nova still needs.

## What This Vision Requires

### 1. Project Workspaces
Nova needs project containers that can hold:
- repo/folder
- linked docs
- project memory
- reports
- trusted sources
- active tasks

### 2. Read And Write Workflows
Nova should eventually be able to:
- read docs/code
- update documents
- prepare GitHub changes
- create reports
- keep project materials current

But governed write/update actions must stay explicit and reviewable.

### 3. Strong Connector Strategy
Nova should connect first to the sources people really use for coding and writing:
- GitHub
- local repos
- VS Code / workspace folders
- Visual Studio solutions
- JetBrains projects
- Cursor projects
- text/markdown files
- Word docs
- Google Docs
- Notion
- Obsidian
- PDFs and references

The source-first rule still applies:
- read-only and workspace-level access first
- app-control and write actions later under stronger governance

### 4. Schedules And Ongoing Work
Nova should eventually support:
- daily brief
- scheduled project summaries
- recurring checks
- reminders
- alert-style tasks

These should stay:
- user-created
- visible
- editable
- pausable
- deletable

### 5. Home Integration
Home integration belongs in the vision, but under strong trust rules.

The safe order is:
- read-only home/device status first
- explicit approved actions second
- no silent environmental action drift

### 6. Personality Stack
Nova should eventually have:
- a stable recognizable identity
- a clear personality stack
- friendly but restrained conversational behavior
- consistency across chat, work, and action surfaces

But personality should not be confused with:
- authority
- hidden adaptation
- silent behavioral drift

### 7. Learning And Adaptation
Eventually Nova may:
- remember durable preferences
- improve retrieval choices
- adapt how it helps
- refine its long-term behavior

But that should happen only under:
- reviewability
- explicit control
- rollback
- auditability

## Phase Placement Map

### Current Product Track
These belong in the current bounded product and system-building lane:
- project/workspace system
- stronger conversation flow
- explicit memory system
- answer-first retrieval and research
- read-only connectors
- evals/traces/regressions
- mode architecture
- trust-center design
- schedule/task design surfaces

Why:
- they build the Agent OS foundation
- they improve robustness now
- they do not require unsafe autonomy

### Phase 7 - Governed External Reasoning
These belong in Phase 7:
- stronger reasoning providers
- second-opinion reviews
- harder synthesis
- deeper research quality

Why:
- they improve the intelligence layer
- they remain text-first
- they do not add execution authority

### Phase 8 - Governed Execution And Real-World Effect
These belong in Phase 8:
- governed document updates
- governed GitHub write actions
- governed app actions
- governed home actions
- approval-gated connector writes
- scheduled actions with real-world effects

Why:
- these are action-capable workflows
- they require explicit approval and trust loops

Important boundary:
- reading and reasoning can start earlier
- actions and writes belong here

### Phase 9 - Continuous Node / Multi-Client Coherence
These belong in Phase 9:
- one continuous Nova across devices/clients
- shared project continuity across surfaces
- shared tasks and schedules across surfaces
- cross-client memory coherence
- stable connector behavior across interfaces
- one trust model across the node

Why:
- this is where Nova becomes truly continuous as a system, not just a local app

### Phase 10 And Later - Reviewable Learning And Adaptive Identity
Only much later should Nova consider:
- durable adaptive learning
- project-specific long-term adaptation
- reviewable personality refinement
- retrieval preference learning
- reduced-friction autonomy in bounded areas

This should happen only with:
- explicit review
- rollback
- diffability
- auditability
- undoability

## What Not To Do
To keep this vision safe, avoid:
- hidden learning early
- silent background action
- authority creep through convenience
- one giant unrestricted always-on agent
- home control without strong trust surfaces
- financial action before serious approval and audit systems

## Best Product Rule
The anchor rule for this vision should be:

Nova should become a continuously available personal agent system by expanding systems, context, and usefulness first, while keeping authority explicit, governed, and reviewable.
