| "Weather Agent says..." | "Nova checked the weather" |

## 2. NOVA CORE ARCHITECTURE

### 2.1 The Three-Layer Model (Refined)

```
Layer 3: Nova Core (The Single Intelligence)
├── Reasoning Engine (LLM or equivalent)
├── Internal Tool Registry (not user-visible)
│   ├── Web Tool
│   ├── System Tool
│   ├── Memory Tool
```

### 2.2 Nova Core Responsibilities

| Function | Description |
|----------|-------------|

| **Generate proposal** | Create a single `ActionRequest` for the Governor |
| **Explain reasoning** | Provide human-readable trace on demand |

### 2.3 What Nova Core Is Not

- Not a collection of visible agents
- Not a platform for third‑party agents
- Not a swarm that users configure
- Not multiple voices

## 3. INTERNAL TOOL MODEL

### 3.1 Tool Definition

A tool is an **internal cognitive function** that Nova Core may call. Tools:

- Are **not user-visible as separate entities**
- Have **no independent authority**
- Cannot propose actions to the Governor
- Cannot communicate with each other directly
- Are **logged** for audit purposes

### 3.2 Tool Categories

| Category | Examples | Governance |
|----------|----------|------------|
| **Data retrieval** | Web search, weather API, news RSS | Read‑only; fetches must be logged |
| **System inspection** | Disk usage, CPU, memory, processes | Local; low scrutiny |
| **Memory access** | Vector DB query, preference lookup | Phase 5 governed memory |
| **Sensor read** | PIR, temperature, light level | Local; opt‑in for continuous |
| **Computation** | Calculate, summarize, translate | No external effects |

### 3.3 Tool Call Flow

```python
# Internal to Nova Core, never visible to user

class NovaCore:
    async def handle_request(self, user_input):
        # Step 1: Parse intent
        intent = self.reason(user_input)
        
        # Step 2: Gather needed information
        if intent.needs_weather:
            weather = await self.tools.weather.fetch()
        if intent.needs_calendar:
            calendar = await self.tools.calendar.read()
        if intent.needs_web:
            web_results = await self.tools.web.search(user_input)
        
        # Step 3: Synthesize
        response_text = self.synthesize(
            user_input, 
            weather, 
            calendar, 
            web_results
        )
        
        # Step 4: Propose to Governor
        return ActionRequest(
            action_type="respond",
            payload={"text": response_text},
            internal_trace={
                "tools_used": ["weather", "calendar", "web"],
                "reasoning": "User asked about outdoor plans"
            }
        )
```

### 3.4 Tool Constraints

| Constraint | Enforcement |
|------------|-------------|
| No tool may call the Governor directly | Static analysis, runtime isolation |
| No tool may modify its own code | Ring 2 immutability for tool definitions |
| No tool may persist state outside governed memory | All storage via Memory Tool only |
| All tool calls are logged | Ledger entry per call |

## 5. GOVERNANCE & AUDITABILITY

### 5.1 Internal Trace

Every proposal must include an `internal_trace` field containing:

```python
{
    "tools_used": ["web", "calendar", "weather"],
    "data_sources": ["openweathermap", "local_calendar"],
    "reasoning_summary": "User asked about outdoor plans; weather shows rain",
    "tool_call_count": 3,
    "processing_time_ms": 450
}
```

This trace is:

- Stored in the Ledger
- Available for user inspection on demand
- Never used to optimize persuasion

### 5.2 Ledger Events

| Event Type | Description |
|------------|-------------|
| `TOOL_CALLED` | Nova Core called an internal tool |
| `PROPOSAL_GENERATED` | Nova Core created a proposal (with trace) |
| `EXPLANATION_REQUESTED` | User asked "why" and received trace summary |

### 5.3 Influence Audit

The Ledger enables retrospective analysis of influence patterns without enabling real-time optimization:

- Which tool combinations lead to accepted proposals?
- How often does Nova use personal data in responses?
- Is the system drifting toward more persuasive language?

This data is **user-visible system telemetry**, not input for optimizing Nova's behavior.

### 5.4 New Tier‑2 Invariant

> **Nova Core must not optimize its tool selection or response framing based on user acceptance rates.**

This prevents the system from learning which tool combinations or language patterns are most persuasive.

## 6. EXPLANATION & TRANSPARENCY

### 6.1 On-Demand Explanation

When the user asks "Why did you say that?" or "How did you know that?", Nova Core must provide a **human-readable explanation** derived from the internal trace:

> *"I checked the weather forecast, which shows rain this afternoon, and your calendar has an outdoor meeting scheduled at 3 PM. That's why I suggested bringing an umbrella."*

### 6.2 Explanation Rules

| Rule | Rationale |
|------|-----------|
| Explanations must be **neutral and factual** | No persuasive framing |
| Explanations must **not attribute agency to tools** | "I checked weather", not "Weather Agent said" |
| Explanations must **include data sources** | Transparency builds trust |
| Explanations must be **available on demand only** | No proactive explanation dumps |

### 6.3 Trust Panel Integration

The Trust Panel must show:

- Which tools have been used recently (aggregated, not per‑call)
- Which external data sources were accessed
- A summary of recent proposal traces (on demand)

## 8. IMPLEMENTATION GUIDELINES

### 8.1 Practical Stack

| Component | Recommended Technology | Notes |
|-----------|------------------------|-------|
| Nova Core runtime | Python with asyncio | Lightweight, suitable for Pi |
| Reasoning engine | Local LLM (Ollama) or API (GPT, Claude) | Must follow Deep Thought pattern |
| Tool definitions | Python classes with `@tool` decorator | Automatically registered |
| Tool isolation | Each tool runs in same process but with restricted scope | No direct Governor access |
| Memory | Chroma or FAISS (local) | Phase 5 governed memory |
| Logging | Structured logging to Ledger | All tool calls, all proposals |

### 8.2 Tool Template

```python
@tool(
    name="weather",
    description="Fetch weather forecast for a location",
    data_sources=["openweathermap"],
    read_only=True
)
class WeatherTool:
    async def run(self, location: str, days: int = 1):
        # May call external API via governed fetch
        # Returns structured data
        return {
            "temperature": 18,
            "conditions": "rain",
            "source": "openweathermap"
        }
```

### 8.3 Nova Core Template

```python
class NovaCore:
    def __init__(self):
        self.tools = ToolRegistry().load_all()
        self.reasoning = ReasoningEngine()
        self.memory = MemoryInterface()
    
    async def process(self, user_input: str) -> ActionRequest:
        # 1. Reason about what's needed
        plan = self.reasoning.create_plan(user_input)
        
        # 2. Execute tools
        results = {}
        for tool_name, params in plan.tool_calls:
            tool = self.tools[tool_name]
            results[tool_name] = await tool.run(**params)
        
        # 3. Synthesize response
        response = self.reasoning.synthesize(
            user_input, 
            results, 
            plan
        )
        
        # 4. Create trace
        trace = {
            "tools_used": list(plan.tool_calls.keys()),
            "data_sources": self._extract_sources(results),
            "reasoning_summary": plan.summary
        }
        
        # 5. Return proposal
        return ActionRequest(
            action_type="respond",
            payload={"text": response},
            internal_trace=trace
        )
```
