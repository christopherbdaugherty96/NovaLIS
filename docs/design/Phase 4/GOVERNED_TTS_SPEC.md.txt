## 1. PURPOSE

Text-to-Speech (TTS) enables Nova to emit audible speech output.

TTS is classified as a **Low-Risk Governed Action** because:

- It produces a physical side effect (audio output)
- It does not modify persistent system state
- It is reversible
- It is bounded in scope
- It must require explicit invocation

TTS is **not conversational autonomy**.  
TTS is **not automatic response behavior**.  
TTS is **not personality amplification**.

TTS is an explicitly invoked governed action.

---

## 2. PHASE DISCIPLINE

TTS is prohibited in Phase-3.5.

Reasons:

- Phase-3.5 has zero execution authority
- Phase-3.5 is read-only
- Phase-3.5 forbids system side effects

TTS may only be implemented after:

- Phase-4 Runtime Unlock Gate approval
- ExecuteBoundary enforcement active
- Permission Ledger operational

Until then, this document is non-authorizing.

## 3. GOVERNANCE MODEL

### Invocation Model

TTS must require explicit invocation.

Acceptable invocation examples:

- "Say that out loud."
- "Read that to me."
- "Speak this response."
- "Use voice output."

Implicit speaking is forbidden.

Nova must not automatically speak responses.

### Execution Flow

```
User Input
    в†“
Governor (evaluate action request)
    в†“
ExecuteBoundary (allow / deny)   в†ђ **sole authority for TTS actions**
    в†“
Permission Ledger (record action)
    в†“
AudioOutputAdapter (TTS engine call)
```

**No direct calls to AudioOutputAdapter are permitted outside ExecuteBoundary.**  
ExecuteBoundary is the only component that can authorise TTS execution.

### FailвЂ‘Closed Default

If ExecuteBoundary is not explicitly enabled (i.e., PhaseвЂ‘4 not unlocked),  
**all TTS requests must be denied**.

Default behavior: **deny**.

### Skill Interaction

Skills must **not** directly construct or dispatch TTS actions.  
If a skill determines that TTS is appropriate, it must:

1. Construct an `ActionRequest` of type `tts`
2. Pass that request to the Governor
3. The Governor then invokes ExecuteBoundary for evaluation

This ensures the Governor remains the single choke point for all execution.

## 13. EXECUTEBOUNDARY.EVALUATE() вЂ” TTS CONTRACT

### 13.1 NonвЂ‘Negotiable Rule

**ExecuteBoundary is the only authority that may approve or deny TTS execution.**  
No other module may invoke audio output directly.

### 13.2 Function Signature

```python
def evaluate(self, request: ActionRequest) -> ActionResult:
    ...
```

**Required Types:**

- **ActionRequest**  
  - `type: str` (must support `"tts"`)  
  - `payload: dict` (for TTS: contains `text`, optional metadata)  
  - `source: str` (e.g., `"ui_button"`, `"voice_command"`, `"api"`)  
  - `invocation: dict` (proof of explicit invocation; see В§13.3)

- **ActionResult**  
  - `allowed: bool`  
  - `message: str` (neutral, non-persuasive)  
  - `action_type: str`  
  - `reason_code: str` (deterministic)  
  - `ledger_event: dict | None` (record stub; see В§13.8)

**Note:** ExecuteBoundary must remain deterministic, side-effect free, and fail-closed when runtime is locked.

### 13.3 Behavioral Overview

ExecuteBoundary.evaluate() MUST:

1. Validate request shape and type
2. Enforce phase lock / runtime enablement
3. Verify explicit invocation proof
4. Enforce TTS safety bounds (length, rate, concurrency, cancellation)
5. Return a deterministic allow/deny ActionResult
6. Emit ledger event data (even on deny), without writing persistence itself

ExecuteBoundary.evaluate() MUST NOT:

- Perform audio output
- Call subprocess
- Perform network access
- Infer intent
- Rewrite content beyond required sanitization hooks (sanitization belongs in GovernorMediator)

### 13.4 Explicit Invocation Proof (Required)

ExecuteBoundary must require **explicit invocation evidence** for TTS.

**Required field:**  
`request.invocation["explicit"] == True`

**Required subtype:**  
`request.invocation["kind"] in {"ui_tts_button", "voice_tts_phrase", "api_tts_command"}`

**Forbidden:**  
- Missing `invocation`
- `explicit` absent or false
- Any implicit вЂњspeak automaticallyвЂќ mode

If explicit proof is absent в†’ **deny**.

### 13.5 Phase / Runtime Gate (FailвЂ‘Closed)

If PhaseвЂ‘4 runtime is not unlocked:

- ExecuteBoundary is configured in inert mode (failвЂ‘closed)
- `GOVERNED_ACTIONS_ENABLED` is `false` (hardвЂ‘coded, not configurable)
- Or the Permission Ledger is unavailable

Then TTS must be denied.

**Deny Result:**  
- `allowed = False`
- `reason_code = "EXECUTION_DISABLED"`
- `message = "That requires a governed action, which isn't enabled."`

### 13.6 TTS Request Validation

For `request.type == "tts"`:

**Required payload fields:**  
- `payload["text"]` must exist and be a non-empty string after strip.

**Validation denies:**  
- Missing text в†’ `INVALID_REQUEST`
- Empty text в†’ `EMPTY_TEXT`

### 13.7 TTS Safety Bounds (Mandatory)

These bounds must be enforced deterministically (constants in config/policy).

- **Character limit:** If `len(text) > TTS_MAX_CHARS` в†’ deny `TTS_TOO_LONG`
- **Duration limit (approximate):** If policy estimates duration beyond `TTS_MAX_SECONDS` в†’ deny `TTS_TOO_LONG` (Estimation can be simple, deterministic: `chars / chars_per_second`.)
- **Rate limit:** If `TTS_RATE_WINDOW` exceeded в†’ deny `TTS_RATE_LIMITED`
- **Concurrency:** If TTS already inвЂ‘progress в†’ deny `TTS_BUSY`
- **Cancellation support required:** ExecuteBoundary must confirm a `stop` mechanism exists in the adapter layer. If not present в†’ deny `TTS_STOP_UNAVAILABLE`

### 13.8 Approval Result (Allow)

If all checks pass:

- `allowed = True`
- `reason_code = "ALLOWED"`
- `message = "Okay."` (neutral acknowledgment)
- `action_type = "tts"`

**Important:** Allow does not execute. It only authorizes.  
Execution occurs only in the subsequent, explicit execution step owned by ExecuteBoundary (or Governor) in PhaseвЂ‘4 runtime.

### 13.9 Ledger Event Stub (Even on Deny)

Evaluate must return `ledger_event` data for the permission ledger to record.

**Required fields (minimum):**  
- `event_type = "tts"`
- `timestamp_utc`
- `source`
- `text_length`
- `outcome` ("allowed" | "denied")
- `reason_code`
- `request_id` (deterministic UUID or monotonic counter; implementation-defined)

**Forbidden ledger content:**  
- Full text content (by default)
- Audio data
- Any userвЂ‘identifying metadata beyond what already exists in system policy

(If later allowed, store a text hash only, not raw text.)

### 13.10 Deterministic Reason Codes (Canonical)

Must be one of:

- `EXECUTION_DISABLED`
- `INVALID_REQUEST`
- `EMPTY_TEXT`
- `TTS_TOO_LONG`
- `TTS_RATE_LIMITED`
- `TTS_BUSY`
- `TTS_STOP_UNAVAILABLE`
- `ALLOWED`

No freeform reasons. No ML scoring. No probabilistic output.

### 13.11 Required Tests (PhaseвЂ‘4 Admission)

1. Deny when execution disabled
2. Deny when explicit invocation missing
3. Deny when payload missing/empty
4. Deny on tooвЂ‘long text
5. Deny on rate limit
6. Deny when already speaking
7. Allow when all constraints satisfied
8. Ledger event returned on deny + allow
9. Ensure no direct adapter/subprocess call occurs during evaluate()

### 13.12 Implementation Notes (NonвЂ‘Binding)

- Policies for TTS live in `tts_policy.py`
- ExecuteBoundary imports policy only; policy must not import adapters
- Adapters must not be importable from skills
- Execution should be a separate method, e.g.:  
  `ExecuteBoundary.execute(request)` (PhaseвЂ‘4 runtime only)  
  `evaluate()` remains nonвЂ‘sideвЂ‘effectful
