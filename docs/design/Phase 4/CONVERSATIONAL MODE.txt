3️⃣ CANONICAL RULES FOR IMPLEMENTATION

Rule 1 – Explicit Mode Switching Only

· DeepSeek mode activates only via:
  · Dedicated UI button/icon
  · Explicit command phrase (e.g., “Ask DeepSeek…”)
· No automatic switching, no time‑based triggers, no contextual inference.

Rule 2 – Nova Silence During DeepSeek Mode

· When DeepSeek mode is active:
  · Nova does not receive the user message
  · Nova does not respond
  · Nova does not perform any background processing or analysis
· This guarantees zero background cognition.

Rule 3 – No Execution in DeepSeek Mode

· DeepSeek responses are informational only.
· DeepSeek cannot:
  · Invoke tools
  · Change system state
  · Trigger notifications
  · Influence the Governor
· Any user request requiring execution must be made in Nova mode.

Rule 4 – Conversation Transparency

· All DeepSeek messages are visibly labeled (e.g., “DeepSeek:”).
· No system‑only, hidden, or unlabeled DeepSeek output.
· Audit logs record every DeepSeek invocation with full prompt/response.

Rule 5 – Context Handling (Explicit Choice)

Option A (Safest – Recommended):
Nova does not automatically see DeepSeek responses. They are treated as a separate conversation thread. If the user wants Nova to consider the DeepSeek output, they must explicitly reference it in a Nova‑mode message.

Option B (Still Safe):
Nova sees DeepSeek responses as normal chat history, but they carry no privileged status – they are treated exactly like user text. Nova may reference them only when explicitly asked.

Our recommendation: Option A for maximum isolation and determinism.
