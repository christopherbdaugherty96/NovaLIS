## Core Read

Nova's durable differentiator is not raw capability count. It is governed trust:

- explicit authority boundaries
- auditable execution
- local-first operation
- visible user control over memory and external effects

That differentiator is real enough now to matter.

---

## Current Strategic Problem

Nova is no longer mainly asking:

> Can this architecture be built?

It is now asking:

> Can a new user install it, understand it, trust it, and feel value quickly?

The next challenge is product legibility, not just technical depth.

---

## Strategic Priorities

### 1. Complete trust-surface basics

Before deeper expansion, Nova still needs:

- clean installer validation
- cap 64 live signoff and lock
- a smoother first-run experience

### 2. Show one undeniable user outcome

The strongest near-term user-value surfaces are:

- continuity across sessions
- safe local email drafting
- governed assistant behavior users can inspect

### 3. Reduce the gap between sophistication and comprehension

Nova already contains meaningful architecture. The current risk is that a new user sees complexity before they see usefulness.

That means:

- simplify first impressions
- keep the first-run path concrete
- center a hero workflow

### 4. Keep architecture subordinate to visible value

The builder instinct will keep wanting:

- deeper memory systems
- more automation
- more adapters and execution lanes

Those may matter later, but the present leverage is still:

- installability
- clarity
- trust
- continuity visibility

---

## What This Memo Recommends

If Nova is choosing what matters most in the next bounded stretch, the priority order should remain:

1. Installer validation
2. Cap 64 closure
3. Demo / README proof
4. First-run clarity
5. Continuity visibility

This prioritization is consistent with the active roadmap in `Now.md` and with the current continuity direction captured in `NOVA_MEMORY_CONTINUITY_UPGRADE_CANDIDATE_2026-04-20.md`.

Everything else should justify itself against those surfaces.

---

## Strategic Caution

Nova should not confuse "more system depth" with "more product readiness."

The risk is not that Nova lacks architecture.

The risk is that Nova can keep building deeper systems before users ever experience the current strengths clearly enough to care.
