## 🔹 Executive Summary  

Phase 6 introduces **delegated autonomy** – the first time Nova can perform actions automatically on behalf of the user, **but only within explicit, user‑defined policy envelopes**. This is not general‑purpose autonomy; it is **bounded, policy‑driven execution** that remains fully governed by the Governor.  

- **User‑Defined Policies:** Users can create rules like “Every morning at 8am, summarise my calendar” or “If motion detected after 10pm, turn on night light.”  
- **Bounded Automation:** Nova monitors conditions and executes actions – **only if** the action is permitted by an active, user‑approved policy. Background monitoring is strictly limited to trigger evaluation; no speculative reasoning or pre‑planning occurs outside an active evaluation cycle.  
- **Atomic Capability Enforcement:** Policies may reference only atomic capabilities from the Phase 4 registry. Dynamic chaining, orchestration, or composition of multiple capabilities within a single policy is **forbidden**.  
- **Local Optimisation (Class A/B/D Mutations):** With evolution enabled (Tier 3), Nova may adjust retrieval weights, compact memory, and refine routing hints – all reversible, logged, and firewalled. Mutations may **never** modify policy definitions, envelopes, or the capability registry.  
- **Governor as Policy Validator:** The Governor’s role shifts from *“Did the user explicitly request this?”* to *“Is this action permitted by the active delegation policy?”*  

All autonomy remains **opt‑in**, **transparent**, and **fully revocable**. No autonomy without explicit user delegation. No silent background execution. The Governor remains the sole execution choke point, now enforcing policy boundaries.  

Phase 6 is a **design‑only phase** – its components may be implemented in code but remain **runtime‑locked** until Phase 4.5 and Phase 5 are complete, and the Phase 6 admission gate is satisfied. Runtime unlock requires proven household value (≥3 months stable deployment) and explicit user opt‑in.  

---

## 🎯 Phase 6 Goals  

1. **Enable user‑defined atomic policies** for bounded automation – users can define a trigger and a single atomic action.  
2. **Implement policy validation in the Governor** – every autonomous action is checked against the active policy envelope.  
3. **Ensure background monitoring is trigger‑only** – no speculative reasoning, pre‑planning, or decision‑tree expansion outside active evaluation.  
4. **Introduce Class A/B/D mutations** (retrieval weight, memory compaction, routing hint) under Tier 3, with evolution ON and full firewall protection. Explicitly forbid mutation of policy definitions.  
5. **Require deterministic evaluation order** when multiple policies trigger simultaneously – alphabetical by policy ID, no prioritisation.  
6. **Design emergency stop that preempts all external timers and listeners** – not merely blocks execution.  
7. **Maintain all Tier 1 invariants** – no self‑expansion of authority, no background cognition without policy, no predictive preloading, no silent fallback.  
8. **Prove household value** – Phase 6 runtime unlock only after ≥3 months of stable deployment with documented daily usage and positive retention metrics.

### ❌ Excluded (Remain Runtime‑Locked or Future)  

- Any form of autonomous cognition without explicit policy (Tier 4 remains Phase 9+).  
- Multi‑step or orchestrated policies (e.g., “if X then A then B”).  
- Mutation of policy definitions, envelopes, resource limits, or capability registry entries.  
- Learning or adaptation that affects constitutional components (Ring 1 immutable).  
- Multi‑user support (remains single‑user).  
- Any capability beyond Phase 4 + Phase 5 memory + Phase 6 atomic policies.  
- Predictive preloading, speculative execution.  
- Emotional simulation or personality drift.

## 🏛️ Architectural Overview  

Phase 6 extends the Governor with a **Policy Validation Engine** and enables background monitoring for active policies. All mutations are mediated by the **Governor Mutation Firewall**.  

```
User-defined Atomic Policies (Ring 3)
    ↓
Governor Policy Validator
    ↓
Background Monitor (Trigger-Only)
    ↓
Governor (evaluate: is action allowed by policy? deterministic order)
    ↓
ExecuteBoundary → Atomic Action (if allowed) → Ledger
```

### Policy Validation Flow  

1. **Policy Creation:** User defines policy via UI or text (e.g., “every weekday at 8am, run morning routine”). Governor validates that the action is a single atomic capability from Phase 4 registry, checks syntax, and stores policy in Ring 3.  
2. **Policy Activation:** User enables policy (explicit toggle). Governor registers the trigger with the background monitor: for time‑based, uses external OS timers; for event‑based, subscribes to device agent events (e.g., motion sensor).  
3. **Background Monitoring (Trigger‑Only):** When a trigger occurs, the monitor invokes Governor with the policy ID and current context. **No pre‑processing, no speculative reasoning** – only trigger detection.  
4. **Action Evaluation:** Governor checks:  
   - Is policy still enabled?  
   - Is the action within envelope (atomic capability, resource limits)?  
   - If multiple policies triggered, evaluate in deterministic order (alphabetical by ID).  
5. **Execution:** If all checks pass, Governor invokes the capability via ExecuteBoundary, logging the autonomous action.  
6. **Emergency Stop:** User hits stop → Governor immediately unregisters all timers/listeners, cancels pending triggers, disables all policies, and logs the event.  

### Mutation Framework for Class A/B/D  

- All mutations are expressed as `MutationRequest` and pass through the Governor Mutation Firewall (see [Firewall & Ledger Specification](Firewall%20%26%20Ledger%20Specification.txt)).  
- Mutations are allowed only if:  
  - Evolution Kill Switch = ON.  
  - Autonomy Tier ≥ 3.  
  - Target component is `@mutable` and in Ring 2.  
  - Delta limits and frequency caps are satisfied.  
  - After‑state simulation matches claimed state.  
- **Mutations may never modify policy definitions, policy envelopes, or capability registry entries.**  
- All mutations are logged; rollback is possible via stored snapshots.

## ⚠️ Risks and Mitigations  

| Risk | Mitigation |
|------|------------|
| **Policy loopholes (unintended actions)** | Governor validates every action against atomic policy; only single capability allowed. |
| **Background monitoring becomes speculative** | Strictly trigger‑only; no pre‑processing; all monitoring logged. |
| **Mutation destabilises system** | All mutations reversible; delta limits; simulation before commit; rollback on threshold exceeded. |
| **User creates dangerous atomic policies** | Policy UI includes warnings; Governor blocks actions outside envelope; emergency stop always available. |
| **Multi‑policy race conditions** | Deterministic alphabetical order; no prioritisation. |
| **Policy self‑mutation** | Mutation firewall explicitly forbids mutations to policy definitions. |
| **Emergency stop fails to stop external timers** | Stop unregisters timers/listeners; verified in tests. |

## ✅ Success Criteria for Phase 6 Design Completion  

- Atomic policy framework implemented and tested: users can define, enable, disable, delete policies with single atomic actions.  
- Governor correctly evaluates actions against policies; blocks any outside envelope; enforces deterministic order.  
- Background monitor triggers policies via external timers/events; no speculative reasoning.  
- Mutation firewall implemented for Class A/B/D; mutations logged, reversible, and respecting limits; policy mutations blocked.  
- Emergency stop unregisters all external triggers and disables policies within 100ms.  
- UI for policy management and activity log passes Four Gates.  
- Adversarial tests show no bypass; constitutional audit confirms no authority expansion, no orchestration.
