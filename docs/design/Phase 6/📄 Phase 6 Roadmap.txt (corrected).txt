# Phase 6 Roadmap (Corrected)
Date: 2026-03-13
Status: Planning packet only; runtime not authorized
Scope: Ordered roadmap for the Phase-6 core package after Phase-5 closure

## Current Starting Point
Phase 5 is now closed for the current trust-facing package.

That means Nova already has:
- governed memory
- session-scoped project continuity plus durable memory bridge
- memory inspectability
- manual tone controls
- user-directed scheduling
- opt-in pattern review

Phase 6 begins only where Nova may perform bounded automatic action inside explicit policy envelopes.

## Core Interpretation Rule
Phase 6 is not "more helpful automation" in the abstract.
It is the first phase where Nova may move from:
- user invocation only

to:
- policy-bound delegated action

But only if every action remains:
- explicitly delegated
- atomic
- Governor-validated
- logged
- revocable
- emergency-stoppable

## What Phase 6 Is
Phase 6 is the design and future implementation path for:
- atomic policy language
- policy envelopes
- Governor policy validation
- trigger-only monitoring
- deterministic policy evaluation order
- emergency stop
- transparency and activity logging for policy-driven execution

## What Phase 6 Is Not
Phase 6 is not:
- general autonomy
- multi-step orchestration
- autonomous cognition without policy
- hidden reminders or inferred delegation
- silent authority expansion
- a reopen of deferred Phase-5 tracks by default

## Correct Phase-5 -> Phase-6 Handoff
Phase 5 hands Phase 6 three important things:
1. explicit persistence without hidden memory drift
2. explicit user-directed scheduling without autonomous execution
3. a closed trust model that must remain intact

That means the Phase-6 core must build on top of:
- explicit user settings
- Governor mediation
- policy inspectability
- quiet, non-persuasive UX

It must not collapse Phase 5's trust-facing surfaces into hidden automation.

## Ordered Phase-6 Roadmap

### Milestone 0 - Phase-5 Alignment Freeze
Purpose:
- lock the Phase-5 boundary before any delegated-autonomy work begins

Required outcome:
- Phase-5 closure remains the authority for the current runtime
- deferred Phase-5 tracks stay deferred unless separately approved
- current runtime remains non-autonomous

Inputs:
- `docs/PROOFS/Phase-5/PHASE_5_CLOSED_ACT_2026-03-13.md`
- `docs/PROOFS/Phase-6/PHASE_6_DEFERRED_FROM_PHASE_5_2026-03-13.md`

### Milestone 1 - Atomic Policy Foundation
Purpose:
- define the smallest lawful unit of delegated action

Required work:
- policy language for one trigger -> one atomic action
- policy envelope schema
- policy storage model
- explicit enable/disable model

Current core design artifact:
- `docs/design/Phase 6/ATOMIC_POLICY_LANGUAGE_AND_POLICY_ENVELOPE_SPEC.md`

Why first:
- nothing else in Phase 6 is safe until "what a policy is" is fixed

### Milestone 2 - Governor Policy Validator
Purpose:
- make the Governor the sole authority for delegated execution

Required work:
- Governor policy validation path
- atomic-capability allowlist checks
- envelope/resource checks
- explicit block reasons
- emergency stop semantics

Why second:
- trigger monitoring cannot be allowed to execute anything until the Governor can validate every policy action

### Milestone 3 - Trigger-Only Monitoring
Purpose:
- allow bounded background evaluation without background cognition drift

Required work:
- time/event trigger system
- strict trigger-only monitoring rules
- deterministic evaluation order
- no speculative planning
- no multi-policy orchestration

Why third:
- after the policy model and Governor validator exist, monitoring can remain a thin trigger surface instead of becoming a second brain

### Milestone 4 - Transparency, Ledger, and User Control
Purpose:
- make delegated behavior inspectable enough to trust

Required work:
- activity log for policy evaluations and outcomes
- policy creation/edit/delete history
- explicit enable/disable visibility
- last-triggered / next-triggered surfaces
- emergency-stop visibility

Why fourth:
- delegated operation without strong inspection would violate Nova's existing trust model

### Milestone 5 - Policy UI and Everyday Use Surfaces
Purpose:
- give users a clear way to author and manage policies without persuasion or dark patterns

Required work:
- policy list view
- policy editor
- policy status surfaces
- neutral explanations of why a policy was allowed or blocked

Why fifth:
- by this point the underlying rules are fixed, so the UI can reflect them honestly instead of inventing behavior

### Milestone 6 - Constitutional Verification and Runtime Gate Review
Purpose:
- verify that the delegated model still preserves Agent Under Law

Required work:
- constitutional audit
- adversarial tests
- emergency-stop tests
- policy-bypass tests
- final unlock decision packet

Why last:
- closure has to evaluate the real assembled system, not isolated parts

## Correct Ordering Rule
If there is any conflict between delegated policy execution and future mutation/evolution theory, the delegated policy core comes first.

In other words:
1. define policy
2. validate policy in Governor
3. monitor triggers safely
4. expose inspection and emergency stop
5. verify constitutionality
6. only then consider whether mutation/evolution work should be pulled forward

## What Is Deferred, Not Automatically Added
The following do not automatically become Phase-6 core just because they were deferred from Phase 5:
- declarative identity/preferences
- persistent cross-session thread substrate / restoration layer
- broader tone-calibration evolution beyond manual controls

They remain candidate tracks and require their own explicit specification path.

## Admission Reality
Phase 6 should still be treated as runtime-locked until its own admission conditions are satisfied.

Current practical interpretation:
- planning may continue
- implementation ordering may be documented
- runtime unlock is not implied by this roadmap
