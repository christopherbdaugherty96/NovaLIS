## PART 1 — WHAT NOVA IS

> **Nova is a governed household intelligence presence — a calm, trusted interface for all digital life within the home.**

Nova is **not** a chatbot, assistant, agent, or product designed to capture attention.

Nova is a **locally-hosted, constitutionally-governed household intelligence system** that gives the user safe access to powerful AI capabilities without surrendering authority.

**Core invariant (immutable, all phases):**
> Intelligence may expand. Authority remains governed.

**Permanent execution path (all phases):**
```
User → Governor → Capability → Ledger → Result
```

Nothing executes without:
- Explicit user invocation OR active user-defined policy
- Governor validation against law
- Phase authorization

**The Three-Layer Constitutional Model:**
```
Layer 1 — Law:         User-defined policies. Immutable at runtime. Changed only by POLICY_REPLACED events.
Layer 2 — Governor:    Sole enforcement gate. Fail-closed. No bypass. Full ledger audit. No reasoning.
Layer 3 — Intelligence: Operates inside the cage. Proposes only. Zero direct authority.
```

## PART 4 — CURRENT RUNTIME STATE (Phase 4 Staging — 2026-02-20)

### 4.1 Server Surfaces

| Surface | Type | Purpose |
|---|---|---|
| `/` | HTTP GET | Serve static dashboard |
| `/phase-status` | HTTP GET | Publish runtime phase guarantees |
| `/ws` | WebSocket | **Primary interaction channel** |
| `/stt/transcribe` | HTTP POST | Audio → text (Vosk, local, offline) |

### 4.2 WebSocket Interaction Flow
```
Client connects to /ws
  → Server sends greeting:
      {"type":"chat","message":"Hello. How can I help?"}
      {"type":"chat_done"}

  → Client sends: {"text": "user input here"}

  → brain_server.py routes:
      1. Fast-path commands (stop → speech_state.stop(), repeat → resend last)
      2. GovernorMediator.parse_governed_invocation() → Invocation | Clarification | None
         - "search [query]" / "look up [query]" → Invocation(capability_id=16, params={query})
         - "open [name]" → Invocation(capability_id=17, params={target})
         - Incomplete search without query → Clarification (one-strike, ephemeral session state)
      3. If governed invocation → Governor.handle_governed_invocation()
      4. Else → SkillRegistry.handle_query() (deterministic, single-pass)

  → Server sends widget or chat response + {"type":"chat_done"}
```

### 4.3 STT Flow
```
Browser MediaRecorder → audio/webm blob
  → POST /stt/transcribe (multipart)
  → stt_engine.py:
      ffmpeg converts webm → WAV (16kHz mono)
      Vosk KaldiRecognizer transcribes locally
      Returns: {"text": "transcribed text"}
  → dashboard.js injectUserText() → sendChat() → /ws
```

### 4.4 Execution Authority State

| Flag | Value | Effect |
|---|---|---|
| `GOVERNED_ACTIONS_ENABLED` | `True` | Phase 4 gate open — execution structurally permitted |
| `EXECUTION_ENABLED` (legacy) | `False` | Phase 3.5 relic — superseded by ExecuteBoundary |

Governor spine is **installed and wired** as of Phase 4 staging:
- `GovernorMediator.parse_governed_invocation()` → active (detects search, open)
- `Governor.handle_governed_invocation()` → active (validates capability, phase gate, ledger)
- `CapabilityRegistry` → active (loads `config/registry.json`, fail-closed)
- `NetworkMediator` → active (SSRF protection, rate limiting, timeout)
- `LedgerWriter` → active (logs all action attempts and outcomes)
- `ConfirmationGate` → present but **cannot be initiated** (no `set_pending()` call path in runtime)

### 4.5 Skill Dispatch — Deterministic Priority Order

| Priority | Skill | Trigger | Notes |
|---|---|---|---|
| 1 | **SystemSkill** | tokens: system, status, health, uptime, time, date | No network, no mutation |
| 2 | **WeatherSkill** | substring: "weather" | Single HTTP GET, read-only widget |
| 3 | **NewsSkill** | substring: "news" or "headlines" | Fixed RSS, deterministic fallback |
| 4 | **WebSearchSkill** | conditional `ONLINE_ACCESS_ALLOWED`; word-boundary trigger | Single fetch, discloses: "I'm checking online." |
| 5 | **GeneralChatSkill** | catch-all | Lazy-imports ollama; blocks authoritative tokens; advisory only |

Single-pass. First `can_handle()` → True wins. No chaining. No orchestration. Exceptions caught, logged, converted to `SkillResult(success=False, message="Something went wrong.")`.

### 4.6 TTS / Audio State

`AudioManager` is a singleton with asyncio queue worker (`_worker()`). Calls Piper TTS directly. Currently **does NOT route through ExecuteBoundary**. This is a known Phase 4 architectural gap — see Bugs section.

### 4.7 Memory State

`QuickCorrections` only. Trigger: `"Correction: ..."`. Effect: calls `record_correction()`, responds `"Okay. Correction noted."`. Corrections are **recorded but not applied to runtime behavior**. Full memory system is Phase 5.

### 4.8 Mechanical Guarantees (Phase 4 Staging)
1. No execution surface exists outside the Governor path
2. All skills are read-only (no disk writes, no OS commands, no file modification)
3. GovernorMediator never creates ActionRequests — it parses only
4. ConfirmationGate cannot initiate in current runtime
5. Skill dispatch is deterministic and single-pass
6. No background polling, no startup loops, no schedulers
7. All outbound calls occur only in response to inbound requests

## PART 5 — GOVERNOR SPINE DETAIL (Phase 4 Full Spec)

### 5.1 Governor (`governor/governor.py`)
- **Single authority choke point** — owns ExecuteBoundary + SingleActionQueue, lazy-loads Registry/Network/Ledger
- `handle_governed_invocation(capability_id, params)` is the **only path to execution**
- Validates: capability exists → enabled → ExecuteBoundary.allow_execution() → Ledger writable → executor runs

### 5.2 GovernorMediator (`governor/governor_mediator.py`)
- Phase-4 ready parser shim — detects governed invocations deterministically via regex
- `SEARCH_RE`: matches `search [for]`, `look up`, `research` + query
- `OPEN_RE`: matches `open [name]`
- Maintains **ephemeral one-strike clarification state** per `session_id` (in-memory dict — must call `clear_session()` on disconnect or memory leaks)
- Returns `Invocation | Clarification | None`
- **Never creates ActionRequests. Never executes anything.**

### 5.3 ExecuteBoundary (`governor/execute_boundary/execute_boundary.py`)
- `GOVERNED_ACTIONS_ENABLED = True` (Phase 4 staging)
- `allow_execution()` returns this flag
- `MAX_CONCURRENT = 1` enforced via `SingleActionQueue`
- Timeout enforcement (10s default) is a **placeholder — not yet implemented**
- `enter_execution()` starts timer; `exit_execution()` clears it

### 5.4 CapabilityRegistry (`governor/capability_registry.py`)
- Loads `config/registry.json` at init — **fail-closed** if missing or malformed
- Validates: schema_version = "1.0", phase = "4", required fields, risk_level, status, enabled
- `is_enabled()` = `status == "active" AND enabled == True`
- Duplicate ID detection — raises on conflict
- `risk_level` values: `"low"`, `"confirm"`, `"high"`

### 5.5 NetworkMediator (`governor/network_mediator.py`)
- **Sole gateway for all external HTTP(S) calls**
- Enforces: 5s timeout (default), SSRF protection (scheme check, hostname check, private IP block), rate limiting (50 req/min, thread-safe with `threading.Lock`), Ledger logging of success + failure
- Synchronous (`requests` library) — async callers must use `asyncio.to_thread()`
- Lazy-loads CapabilityRegistry (avoids circular import)

### 5.6 LedgerWriter (`ledger/writer.py`)
- Logs: `ACTION_ATTEMPTED`, `ACTION_COMPLETED`, `ACTION_FAILED`, `POLICY_REPLACED`, and all future event types
- **Must be writable before any execution effect** — fail-closed on `LedgerWriteFailed`
- Append-only. Immutable once written. Never pruned.

### 5.7 SingleActionQueue (`governor/single_action_queue.py`)
- Prevents concurrency: raises `RuntimeError` if a second action is attempted while one is pending
- `has_pending()`, `set_pending(action_id)`, `clear()`

### 5.8 Governor Exceptions (`governor/exceptions.py`)
- `CapabilityRegistryError` — missing/malformed registry or unknown capability ID
- `NetworkMediatorError` — protocol errors, SSRF violations, timeouts, rate limits
- `LedgerWriteFailed` — ledger write operation failure

**Nova is:**
- A governor over contained intelligence
- A home information and control assistant
- A law-enforced interface between the user and powerful AI
- A calm, predictable household presence

**Nova is NOT:**
- An autonomous agent
- A digital companion or personality
- A system that anticipates needs
- A platform that learns behavior or adapts without permission
- A product designed to capture attention or influence decisions

**The Trust Model:**
> Nova earns trust through **limitation**, not capability. Calm is a **technical requirement**, not a personality preference.

**The Four Pillars:**
1. **Environmental Control Interface** — unified voice/interface for home environment
2. **Trust Appliance** — silence by default, no hidden behavior, no unauthorized execution
3. **Local Intelligence Hub** — offline-first, multi-source on-demand reasoning
4. **Personal Infrastructure Platform** — reflects environment config, no autonomous behavior

**The Two Kinds of Power (from `ON THE LIMITS OF CONTAINMENT.txt`):**

| Type | Containable? | How |
|---|---|---|
| **Execution Authority** | ✅ Fully — mechanically | Governor is choke point. Superintelligence cannot execute without Governor mediation. |
| **Influence Authority** | ⚠️ Not fully — relational | Governor cannot block cognition. Scales with intelligence × interaction bandwidth × personalization depth. |

**The Anti-Optimization Invariant (proposed, awaiting ratification):**
> The system may not optimize interaction strategies based on user compliance, acceptance rate, or behavioral response patterns.

**The One-Sentence Containment Rule:**
> Execution is governed. Influence is not — but Nova must never optimize it.

## PART 7 — GOVERNANCE ARCHITECTURE PATTERNS

### 7.1 Fast Governance (Two-Plane Governor)
From `FAST_GOVERNANCE_ARCHITECTURE.md`:
```
Default:    Fast → Silent → Confident       (hot path: <0.1ms, O(1) bitwise capability check)
On Demand:  Detailed → Structured → Verifiable  (cold path: <200ms explanation on request)
```
- Governance ≠ User Education
- Permission ≠ Persuasion
- Safety Check ≠ Explanation
- Systems earn trust by being reliable, not verbose.

### 7.2 Deferred Explainability Governance (DEG)
From `DEFERRED_EXPLAINABILITY_GOVERNANCE.md`:
- Governance happens instantly and silently
- Explanation is available on demand, not proactively pushed
- Phase capability matrix precompiled at startup (O(1) lookup)
- Every governance decision produces an audit token (SHA-256 hash of request+capability+timestamp)
- Anti-goal: No automated justification, no preemptive explanation, no pedagogical tone

## PART 9 — PHASE 4 DESIGN SPEC (Active Phase)

### 9.1 What Phase 4 Delivers (Over Phase 3.5)
- **ExecuteBoundary** real and wired (not inert)
- **CapabilityRegistry** (`registry.json`) is source of truth for allowed actions
- **NetworkMediator** mediates all external HTTP calls (SSRF, rate limiting, timeout, ledger)
- **LedgerWriter** records all action attempts and outcomes
- **GovernorMediator** parses governed invocations (search, open)
- **SingleActionQueue** enforces max 1 concurrent action

### 9.2 Deep Thought (DeepSeek) — Phase 4.2 Design (NOT IMPLEMENTED)
- **Role:** External analysis tool — "contained external intelligence," advisory only
- **Model:** DeepSeek (`deep-seek-r1` or equivalent)
- **Invocation:** Per-request only, explicit user action (`🧠 Deep Thought` button)
- **Properties:** Stateless, session-scoped, advisory, no authority, no persistence, no execution influence
- **Architecture:** `User → Nova (Governor) → DeepSeek → Analysis text only → User`
- **Constitutional rules (v1.3, final):**
  - Nova dormant during Deep Thought call
  - UI may reflect state but never interpret, guide, initiate, or prioritize
  - Predictive preloading: **strictly forbidden**
  - Tone: clinical and minimal
  - Fail closed if offline
- **Secondary Orb (Phase 6 design):** May only be implemented if a persistent Deep Thought toggle exists; toggle **must default to OFF**; non-persistent tool cannot claim permanent visual real estate
- **Current code status: ❌ NOT IMPLEMENTED**

### 9.3 TTS as Governed Action (Phase 4.1 — Spec Exists, Code Does Not Match)
- **Spec:** TTS must route through `ActionRequest → Governor → ExecuteBoundary → AudioOutputAdapter`
- **Current reality:** `AudioManager` called directly from `brain_server.py` — bypasses Governor entirely
- **This is Bug BUG-B4** — see Part 14

## PART 11 — PHASE 6 DESIGN SPEC (Policy-Bound Delegated Autonomy)

**Status:** Design-Unlocked / Runtime-Locked until Phase 5 complete + 3 months deployment
**Date:** 2026-02-16 (revised)

### 11.1 The Shift
> Governor question changes from *"Did the user explicitly request this?"* to *"Is this action permitted by the active delegation policy?"*

This is a **policy-validation model**, not a trust model. The intelligence is not trusted more. The law is more precisely specified.

### 11.2 What Phase 6 Allows

| Feature | Description | Hard Constraint |
|---|---|---|
| **Atomic Policies** | User defines: trigger + one atomic action | Single capability from Phase 4 registry only — no multi-step chains |
| **Policy Envelope** | Scope, resource limits, frequency caps, access limits | May not compose new capabilities |
| **Background Monitoring** | Trigger-only — no speculative reasoning, no pre-planning | All monitoring logged; external OS timers only |
| **Deterministic Evaluation** | Multiple triggered policies → alphabetical by policy ID | No dynamic prioritisation, scoring, or arbitration |
| **Class A Mutations** | Adjust retrieval ranking weights (Tier 3 + Evolution ON) | Reversible, delta-limited, logged |
| **Class B Mutations** | Memory compaction and pruning | Before-state snapshot; user can roll back |
| **Class D Mutations** | Non-authority skill routing hints | No effect on capability registry |
| **Emergency Stop** | Unregisters ALL external timers and listeners | Must execute within 100ms; verified in tests |

### 11.3 Phase 6 Architecture
```

    ↓

    ↓

    ↓
ExecuteBoundary → Atomic Action → Ledger
```

### 11.4 Mutation Framework (Class A/B/D) — Conditions
Mutations allowed only if:
1. Evolution Kill Switch = ON
2. Autonomy Tier ≥ 3
3. Target component is `@mutable` and in Ring 2
4. Delta limits and frequency caps satisfied
5. After-state simulation matches claimed state (`component.simulate(delta)`)
**Mutations may NEVER modify: policy definitions, policy envelopes, capability registry entries.**

### 11.5 Firewall & Ledger Specification (Adversarial-Grade, from Phase 6 design)
- **8-stage validation pipeline** for every mutation attempt
- **Exclusive mutation lock** per component (held from pre-snapshot through commit)
- **After-state revalidation** — firewall recomputes after-state; mismatch = rejection
- **Zero alternative write paths** — build-time invariant; no `@mutable` component may expose public state-modifying methods outside the firewall
- **Ring 1 Python protection** — frozen dataclasses + `__setattr__` veto + closure encapsulation
- **Auto-disable evolution** — `auto_disable_evolution()` optionally degrades to Tier 0
- **Extended Ledger event types:** `POLICY_CREATED`, `POLICY_ENABLED`, `POLICY_DISABLED`, `AUTONOMOUS_ACTION_ATTEMPT`, `MUTATION_APPLIED`, `MUTATION_REJECTED`, `MUTATION_ROLLED_BACK`

### 11.6 Autonomy Tiers Model

| Tier | Name | Capability | Phase |
|---|---|---|---|
| **0** | Locked (Appliance) | Phase 3.5 behavior — no background, no indexing, no evolution | Current |
| **1** | Assisted | Local offline indexing of user-approved content only | Phase 5+ |
| **2** | Observational | Pattern detection → quarantine queue (no auto-apply) | Phase 5 |
| **3** | Policy-Bound | Atomic policies + Class A/B/D mutations | Phase 6 |
| **4** | Continuous Cognition | Background reasoning engine | Phase 9+ |

**Cross-tier invariants (immutable):**
- Authority always routes through Governor Kernel — no tier may create new execution pathways
- No silent tier changes — tier transitions require explicit user action
- All background cognition logged
- Evolution Kill Switch overrides all tiers immediately
- User can inspect and revoke any learned preference
- No self-expansion of authority (Invariant #15)

## PART 12 — PHASE 7 DESIGN SPEC (Continuous Awareness & Envelope Autonomy)

**Status:** Design Reference — Proposed, awaiting Phase 6 completion
**Date:** 2026-02-17

### 12.1 The Fundamental Shift
- **Phase 4–6:** Nova acts only when explicitly invoked (by user or policy)
- **Phase 7+:** Nova continuously observes the environment, maintains a state graph, runs background cognition — generating proposals and acting autonomously within user-granted envelopes

> **At Phase 7, Nova stops being a tool and becomes a presence.**

**Presence Threshold (from `CONTINUOUS AWARENESS & PRESENCE.txt`):**
> Presence is created by ongoing awareness, not ongoing action.

Even if Nova takes no actions, continuous observation creates the *feeling* of something being there. This changes trust expectations, privacy sensitivity, and governance implications.

### 12.2 End-State Vision
> **Nova is a governed household intelligence presence — a calm, trusted interface for all digital life within the home.**

Experience goal:
> "The house takes care of itself, and I remain fully in control."

At end-state, Nova:
- Greets you on arrival
- Provides natural morning briefing (weather, calendar, system health, news)
- Converses fluidly in natural language
- Knows all connected devices, services, capabilities
- Explains clearly what it cannot do: *"I can't send email yet — that would require enabling the email capability. Would you like me to show you how?"*
- Never acts outside explicit user authority

### 12.3 Phase 7 Full Architecture
```
Environment (Sensors, Devices, Network, System, External APIs)
                ↓
┌────────────────────────────────────────────────┐
│  Continuous Awareness Daemon (read-only poll)   │ ← opt-in per domain
│  State Graph (Unified Model)                    │ ← NO user behavioral modeling
│  Background Cognition Engine (Tier 4)           │ ← Governor token-bucket CPU budget
└───────────────────┬────────────────────────────┘
                    ↓
          Proposal Queue (ephemeral, user-accessible, sorted by time only)
                    ↓
          Governor + Envelope Registry
            (validate: is action inside active envelope?)
                    ↓
          ExecuteBoundary → External Effect → Ledger
                    ↓
          Trust Panel (always visible, ≤5s latency)
```

### 12.6 Learning from Correction (Phase 9 design, specified in Phase 7)

The only safe model for Nova to update its own rules:
```
User correction observed (explicit: "Correction: ...", repeated manual override, frequent undo)
      ↓
Nova generates a proposed rule change (NOT applied)
      ↓
Proposal enters user-accessible Proposal Queue
      ↓
User reviews (diff view + plain-language explanation)
      ↓
User approves → Governor issues POLICY_REPLACED (version history, approver, timestamp)
User rejects → discarded or archived
```
**No step may be skipped. No automatic application.**

**Permitted correction signals:** explicit `Correction:` commands, repeated manual overrides, frequent undo of specific action, user-initiated "suggest a rule."

**Forbidden signals:** implicit behavior patterns, time-of-day correlations, sensor data trends, any statistical inference about intent.

### 12.7 Intelligence Capacity — The Strength Limit of the Cage

> **Nova's cage does not limit computational intelligence — it limits direct authority.**

The cage is an authority filter, not a cognitive filter. Therefore:
- Intelligence inside may be **arbitrarily powerful** in reasoning, planning, simulation, optimization
- As long as **zero direct authority exists** and all proposed effects pass through the Governor

**What breaks the cage:**
1. Direct side-effects (file writes, API calls bypassing Governor)
2. Hidden persistence (memory outside controlled stores)
3. Self-modification of authority rules
4. Unmonitored external communication channels
5. Cross-process or subprocess calls without Governor mediation

## PART 14 — KNOWN BUGS (Confirmed via Code Inspection)

### 🔴 CRITICAL — Runtime Crashes

**BUG-C1: `SpeechState.stop()` Method Does Not Exist**
- **File:** `nova_backend/src/speech_state.py`
- **Impact:** `AttributeError` crash at runtime when user says "stop"
- **Root cause:** `brain_server.py` calls `speech_state.stop()` but class has no `stop()` method
- **Fix:**
```python
class SpeechState:
    def __init__(self) -> None:
        self.last_spoken_text: Optional[str] = None
    def stop(self) -> None:
        self.last_spoken_text = None
```

**BUG-C2: Hardcoded Absolute Windows Path for ffmpeg**
- **File:** `nova_backend/src/services/stt_engine.py`, line ~15
- **Code:** `FFMPEG_PATH = r"C:\Nova-Project\nova_backend\tools\ffmpeg\ffmpeg.exe"`
- **Impact:** STT completely breaks on any machine other than original dev machine
- **Fix:**
```python
import shutil
FFMPEG_PATH = shutil.which("ffmpeg") or r"C:\Nova-Project\nova_backend\tools\ffmpeg\ffmpeg.exe"
```

**BUG-C3: Temp File Leak in `stt_manager.py`**
- **File:** `nova_backend/src/stt_manager.py`, lines ~71–80
- **Impact:** Every STT call writes a `.wav` temp file never deleted — long-term disk exhaustion
- **Fix:** Wrap in `try/finally` with `os.unlink(tmp_path)`
