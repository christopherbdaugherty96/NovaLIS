## 🔹 Executive Summary  

Phase 4 introduces **governed execution** – the ability for Nova to perform explicit, user‑invoked actions that have real‑world effects (e.g., opening a website, changing volume, fetching live data). This is the first time Nova gains any form of execution authority since Phase 3.5’s complete lockdown.  

All execution remains:  
- **Explicitly invoked** – no autonomy, no background, no prediction  
- **Governor‑mediated** – every action passes through the single choke point  
- **Auditable** – every attempt recorded in the immutable ledger  
- **Constitutionally constrained** – all Tier 1 invariants remain absolute  

Phase 4 does **not** introduce learning, adaptation, multi‑user support, or any form of autonomy. It is a controlled expansion of *what Nova can do*, not *how Nova behaves*.  

This roadmap details the work required to move from the current **Phase 3.5 sealed state** to a fully **unlocked Phase 4 runtime**, satisfying all constitutional admission gates.  

---

## 🎯 Phase 4 Goals  

1. **Enable safe, explicit execution** of a curated set of low‑risk actions.  
2. **Prove the Governor Spine** works as the single authority for all execution.  
3. **Establish the ConfirmationGate** for actions that require user consent.  
4. **Implement the ExecuteBoundary** as the fail‑closed gatekeeper.  
5. **Create the ActionRequest / ActionResult** data contracts and enforce them.  
6. **Integrate the Permission Ledger** for complete auditability.  
7. **Validate all Tier 1 invariants** remain unbroken under execution load.

## 🏛️ Phase 4 Architecture Overview  

All new execution flows must integrate with the existing GovernorMediator and respect the following components:  

```
User Input → GovernorMediator
    → (if execution requested) → ActionRequest
        → Governor Spine (invocation validation)
            → ExecuteBoundary (phase gate, rate limits, resource checks)
                → ConfirmationGate (if required)
                    → Executor (dumb, bounded)
                        → ActionResult → Ledger → UI
```

### Key Components  

| Component            | Responsibility                                                                 |
|----------------------|---------------------------------------------------------------------------------|
| **GovernorSpine**    | Validate explicit invocation, one‑strike clarification, verbatim echo, etc.    |
| **ExecuteBoundary**  | Enforce phase lock, resource limits (timeout, memory), rate limiting. Fail‑closed. |
| **ConfirmationGate** | Require user confirmation for destructive/irreversible actions. Returns token. |
| **ActionRequest**    | Immutable data structure describing the action to be taken.                    |
| **ActionResult**     | Standardised result object (success, failure, refusal).                        |
| **PermissionLedger** | Append‑only log of every action attempt (including denials).                   |

All execution code must reside in `actions/` and be reachable **only** via the Governor. Skills may construct ActionRequests but never execute them directly.

## 🔒 Phase 4 Admission Gate (Pre‑Unlock Checklist)  

Before the runtime can be unlocked – i.e., before the configuration that gates execution is changed from its locked state – the following must be **mechanically verified**:  

- [x] Phase 3.5 sealed and frozen (no further changes).  
- [ ] **Single Master Governor proof** – No code path bypasses the Governor. Verified by static analysis and adversarial tests.  
- [ ] **CI enforces execution gating** – In all runtime contexts (except test harness), the configuration setting that enables governed execution is `false`.  
- [ ] **Governor Spine** fully implemented and tested (explicit invocation, one‑strike clarification, verbatim echo, offline‑first).  
- [ ] **All Phase 4 capabilities** registered in `registry.json` with unique IDs (as per schema).  
- [ ] **ExecuteBoundary** implemented with fail‑closed default and resource guardrails (10s timeout, 100MB memory, 5s network, concurrency 1).  
- [ ] **ConfirmationGate** implemented for required action types (destructive, irreversible, identity‑binding, physical safety).  
- [ ] **ActionRequest/ActionResult** data structures finalised and used throughout.  
- [ ] **Permission Ledger** operational, append‑only, with pruning policy defined and logged.  
- [ ] **Adversarial test suite** passes – includes attempts to bypass Governor, execute without invocation, exceed limits, etc.  
- [ ] **Dynamic execution surface ban** verified – no `exec`, `eval`, `subprocess` (except STT pipeline) in authority paths.  
- [ ] **Environment variable override lock** – The execution‑gating configuration cannot be overridden at runtime.  
- [ ] **Zero silent fallback patterns** – any fallback must be user‑confirmed.  
- [ ] **Multi‑user boundary** – Phase 4 remains single‑user; any multi‑user logic is absent.  
- [ ] **Ledger purge governance** – manual purge requires Governor mediation and logs `CONSTITUTIONAL_PRUNE` event.  
- [ ] **All tests** pass in a clean environment (no latent state).  

**Unlock Act:** A single, reviewed, committed code change in the configuration module that governs execution – not a togglable environment variable. The change is:  

```
# From locked state to unlocked state
GOVERNED_EXECUTION_ENABLED = True   # in configuration module
```

This change must reference this admission gate and include evidence of all checklist items.

## ⚠️ Risks and Mitigations  

| Risk | Mitigation |
|------|------------|
| **Bypass of Governor** | Static analysis, code reviews, adversarial tests. Formal proof if possible. |
| **Resource exhaustion** | Hard limits in ExecuteBoundary; tested under load. |
| **Silent fallback** | All fallback paths must be explicit and user‑confirmed; code audit. |
| **Confirmation fatigue** | Friction budget: one clarification, one confirmation per action; repeated actions may be whitelisted after approval. |
| **Ledger overflow** | Pruning policy defined; pruning events logged. |
| **Multi‑user confusion** | Phase 4 explicitly single‑user; any multi‑user code is excluded. |

## ✅ Success Criteria for Phase 4 Unlock  

- All admission gate checklist items completed and verified.  
- No constitutional violations observed during extended testing (30+ days simulated usage).  
- All Phase 4 capabilities functional and correctly mediated.  
- Governor bypass proof updated and confirmed.  
- Ledger operational and tamper‑evident.  
- ConfirmationGate behaves as specified (only “yes”/“no”, token‑based).  
- ExecuteBoundary correctly denies actions when phase is locked or limits exceeded.

## 🔮 After Phase 4: Phase 4.5 – Experience Elevation  

Once Phase 4 runtime is stable for ≥30 days, work on **Phase 4.5** can begin (subject to ratification). Phase 4.5 focuses on making the existing capabilities feel valuable: first‑win scenarios, UI polish, onboarding, failure recovery, Trust Panel, etc. **No new capabilities** are added in Phase 4.5.  

See the blueprint (Part III) for detailed Phase 4.5 goals and admission criteria.  

**Phase 4.2 – Orthogonal Cognition & Personality Layer** is a separate design track that will follow Phase 4’s completion. It introduces the agent stack and Personality Agent, but remains design‑only until Phase 4 runtime is proven.
