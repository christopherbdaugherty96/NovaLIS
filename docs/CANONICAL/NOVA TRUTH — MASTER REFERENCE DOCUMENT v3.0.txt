4.2 WebSocket Interaction Flow
Client connects to /ws
  → Server sends greeting:
      {"type":"chat","message":"Hello. How can I help?"}
      {"type":"chat_done"}

  → Client sends: {"text": "user input here"}

  → brain_server.py routes:
      1. Fast-path commands (stop → speech_state.stop(), repeat → resend last)
      2. GovernorMediator.parse_governed_invocation() → Invocation | Clarification | None
         - "search [query]" / "look up [query]" → Invocation(capability_id=16, params={query})
         - "open [name]" → Invocation(capability_id=17, params={target})
         - Incomplete search → Clarification (one-strike, ephemeral session state)
      3. If governed invocation → Governor.handle_governed_invocation()
      4. Else → SkillRegistry.handle_query() (deterministic, single-pass)

  → Server sends widget or chat response + {"type":"chat_done"}

## PART 12 — PHASE 7 DESIGN SPEC (Continuous Awareness & Envelope Autonomy)
**Status:** Design Reference — Proposed, awaiting Phase 6 completion**Date:** 2026-02-17
### 12.1 The Fundamental Shift
- **Phase 4–6:** Nova acts only when explicitly invoked (by user or policy)
- **Phase 7+:** Nova continuously observes, maintains a state graph, runs background cognition — generating proposals and acting within user-granted envelopes
> **At Phase 7, Nova stops being a tool and becomes a presence.**
### 12.2 End-State Vision
> **Nova is a governed household intelligence presence — a calm, trusted interface for all digital life within the home.**
Experience goal: *"The house takes care of itself, and I remain fully in control."*
### 12.3 Phase 7 Full Architecture
Environment (Sensors, Devices, Network, System, External APIs) ↓ ┌───────────────────────────────────────────────────┐ │ Continuous Awareness Daemon (read-only poll) │ ← opt-in per domain │ State Graph (Unified Model) │ ← NO user behavioral modeling │ Background Cognition Engine (Tier 4) │ ← Governor token-bucket CPU budget └──────────────────────┬────────────────────────────┘ ↓ Proposal Queue (ephemeral, user-accessible, sorted by time only) ↓ Governor + Envelope Registry ↓ ExecuteBoundary → External Effect → Ledger ↓ Trust Panel (always visible, ≤5s latency)

### 12.4 Cognitive Plane Module Map (Phase 7 Implementation)
src/cognitive/ (Plane A — advisory only, never executes) ├── cognitive_plane.py # Main event-driven loop ├── world_model/ │ ├── store.py # Graph + time-series belief store │ ├── updater.py # Ingests events, updates beliefs │ └── query.py # Read-only query interface ├── self_model/ │ ├── capabilities_graph.py # Reads CapabilityRegistry + runtime info │ ├── limits.py # Resource/permission model from ExecuteBoundary │ └── uncertainty.py # Derived from failure logs in Ledger ├── memory/ │ ├── episodic.py # Vector store for events │ ├── semantic.py # Facts/preferences │ ├── procedural.py # User-approved procedures │ └── consolidation.py # Periodic summary ├── planner/ │ ├── planner.py # HTN or tool-augmented planner │ ├── critic.py # Separate model to critique plans │ └── simulation.py # What-if (projects ledger entries, no execution) ├── perception/ │ ├── event_ingest.py # Listens to OS/network/calendar events (opt-in) │ └── filters.py # Decides what to log/update └── proposal_queue.py # Non-executable proposals (Phase 9)

**Cognitive Loop Flow (Phase 7+):**
event_ingest.py: Passively receives allowed events → queue
cognitive_plane.py (every 2s or on event): → world_model/updater.py: Update beliefs → self_model/: Update capability/resource state → Retrieve relevant episodic/semantic memory → planner.py: Generate/critique candidate plans → proposal_queue.py: Push non-executable proposals
Proposal Queue: Read-only surface for user → User says "do that" → Governor converts to ActionRequest → ConfirmationGate → Execute
Governor remains sole executor — Cognitive Plane has zero execution authority
**Constitutional Safeguards:**
- No background execution — Cognitive Plane never calls executors
- No autonomous action — All plans sit in proposal queue
- No silent fallback — If capability unavailable, planner notes it in self-model
- Ledger append-only — Every cognitive update logged
- No behavioral modeling in State Graph (Invariant #26)

### 12.8 Learning from Correction (Phase 9, designed in Phase 7)
User correction observed (explicit: "Correction: ...", repeated manual override, frequent undo) ↓ Nova generates a proposed rule change (NOT applied) ↓ Proposal enters user-accessible Proposal Queue ↓ User reviews (diff view + plain-language explanation) ↓ User approves → Governor issues POLICY_REPLACED (version history, approver, timestamp) User rejects → discarded or archived

**No step may be skipped. No automatic application.**
**Permitted signals:** explicit `Correction:`, repeated manual overrides, frequent undo, "suggest a rule."
**Forbidden signals:** implicit behavior patterns, time-of-day correlations, sensor trends, statistical inference.
