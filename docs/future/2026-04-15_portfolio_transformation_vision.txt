## 4. Long-Term Vision: NovaLIS as Governed Receptionist & Business OS

### The Layered Role Model

| Layer | Role | Examples |
| :--- | :--- | :--- |
| **1. Front Desk** | Answers inbound calls, texts, simple inquiries. | "Thanks for calling Pour Social. How can I help?" |
| **2. Triage Layer** | Classifies inquiry type. | Lead, support, scheduling, urgent, spam. |
| **3. Governed Action Layer** | Only executes explicitly approved actions. | Log details, send summary, draft response, schedule callback. |
| **4. Escalation Layer** | Knows when to hand off to human. | "That requires the owner. I'll have Christopher follow up." |

### Multi-Business Architecture

```

Shared Core Identity (Governance, Ledger, Trust)
├── Pour Social Mode (event inquiries, pricing ranges, availability)
├── Website Business Mode (lead intake, project type, budget, timeline)
└── Personal Mode (reminders, approved contacts, home coordination)

```

### Required Capabilities and Controls

**Capabilities:** Inbound call answering, caller classification, knowledge bases, lead forms, follow-up drafting, call logs, escalation logic, do-not-disturb awareness.

**Controls:** Approved scripts, pricing language, escalation triggers, emergency bypass, business hours, VIP routing, visible history, revocation.

### The Sane Roadmap (Phases 1–5)

| Phase | Focus | Deliverable |
| :--- | :--- | :--- |
| **1** | Local Core Stability | Installable, one strong mutation, memory, basic trust. |
| **2** | Text-Based Business Intake | Form capture, lead summaries, approved responses. |
| **3** | Async Messaging | SMS/WhatsApp/Telegram governed replies. |
| **4** | Voice/Call Handling | Scripted greeting, intake, basic Q&A. |
| **5** | Multi-Business Front Desk | Mode switching, reporting, dashboards. |

### Critical Boundaries

Nova should **never**:
- Promise services not offered.
- Quote unapproved prices.
- Make legal/financial statements outside templates.
- Impersonate the owner without explicit framing.
- Handle issues silently without audit.

> *Nova represents your rules, not its own.*

## 6. Insights from the Claude Leak: Filtered for Practical Use

**Core Principle:** Borrow only patterns that solve Nova's real problems at the right time. Do not worship the leak.

### Concepts Worth Extracting

| Pattern | Description | Applicability | Priority |
| :--- | :--- | :--- | :--- |
| **Dynamic Prompt Assembly / Modular Rules** | Context-specific instructions instead of one giant static prompt. | Cleaner governance. Inject rules only when relevant. | Mid-Term |
| **Mailbox Pattern / Central Approval** | Workers send high-risk requests to a central coordinator. | Natural fit with existing Governor. | Mid-Term |
| **Analytics Dashboard over Raw Logs** | Transform ledger data into insights: failure rates, escalation patterns. | Strategic visibility beyond operations. | Mid-Term |

### Concepts to Defer or Redesign

| Pattern | Caution | Nova Approach |
| :--- | :--- | :--- |
| **"AutoDream" Memory Consolidation** | Autonomous cycles conflict with Nova's values of explicitness and user control. | Manual "optimize memory" command; opt-in scheduled maintenance; visible change summaries. |
| **24/7 Daemon Infrastructure** | Premature expansion distracts from current packaging work. | **Defer.** Complete portfolio upgrade first. |
| **Proactive Intelligence** | Introduces trust and notification complexity before user demand exists. | **Defer.** Wait for clear pull and design bounded controls. |

> *If it works, and it fits—use it. If not, discard it.*
