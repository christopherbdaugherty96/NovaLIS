## What You Should *Immediately* Lock & Implement (Phase 4–5)

These are non-negotiable, low-effort, high-impact safeguards that align with your current trajectory.

### 1. Capability Registry & Lifecycle (Phase 4)
- **Why now:** You're about to integrate sensors, lights, and APIs. Without a registry, you can't track what Nova *can* do, who approved it, or revoke it.
- **Minimal definition:** A simple JSON/YAML file (or DB table) listing every capability (e.g., `light.turn_on`, `weather.fetch`) with fields: `id`, `description`, `required_permissions`, `risk_level`, `enabled_by_default`. The Governor checks against this on every action.
- **Governance:** Adding a new capability requires a user-initiated "install" event (logged). No automatic discovery.

### 2. Model Version Lock (Phase 4.5)
- **Why now:** Swapping a local model or updating an API endpoint changes Nova's behavior silently—a direct violation of your *Evolution vs. Optimization* principle.
- **Minimal definition:** Each model (including external API endpoints) is identified by a version hash. Any update (even a minor prompt tweak) must be treated as a legislative event (`MODEL_UPDATED` in the Ledger) with user confirmation.

### 3. External LLM = Governed Capability (Phase 4)
- **Why now:** If Nova calls an external model (e.g., via API), that's data leaving your home. You must log it and give the user control.
- **Minimal definition:** External model calls are registered as capabilities with a `data_exfiltration` flag. The Governor checks if the user has allowed this specific external service. Log each call (anonymized prompt length, tokens, etc.) but **not** the content (unless required for debugging and explicitly consented).

### 4. Device Authority Scopes (Phase 4.5 – The Missing Piece)
- **Why now:** Once Nova controls physical devices, you need granular permission levels per domain.
- **Minimal definition:** For each capability, assign an `authority_level`:
  - `observe` – can read state, no actions.
  - `suggest` – can generate proposals (Phase 7).
  - `confirm` – can execute, but requires user confirmation each time.
  - `automatic` – can execute autonomously under a token (Phase 6+).
- **UI:** The Trust Panel shows a simple table of domains and their current authority level.

These four additions are **lightweight, enforceable, and directly address real risks** in your current build. They don't require constitutional amendments—just disciplined implementation.
