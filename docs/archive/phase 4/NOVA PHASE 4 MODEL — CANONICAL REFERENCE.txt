### 1.6 Dynamic Execution Surface Ban
**Constitutional Invariant — Hard Locked**

No dynamic execution primitive may be used **outside the Governor’s mediation** or in any component that can influence authority.

This includes, but is not limited to:

- `__import__`, `importlib` (dynamic module loading)
- `exec`, `eval`, `compile`
- `subprocess`, `os.system`, `os.spawn*`
- Reflection‑based method invocation (`getattr` on untrusted names, `__call__` injection)
- Plugin or extension loading not explicitly approved by Governor
- Dynamic code generation or just‑in‑time compilation reachable from authority path

**Exceptions:**
- Static imports (compile‑time resolved) are permitted.
- `importlib` may be used **only** for loading Governor‑controlled, pre‑approved capability modules **after** Phase‑4 unlock, and **only** if the module itself passes Governor proof.
- Any use of dynamic execution primitives **anywhere** in the codebase triggers immediate audit failure and constitutionally blocks Phase‑4 runtime unlock.

**Proof Requirement:**  
The Governor proof must statically and dynamically detect any prohibited dynamic primitive reachable from any entry point that could influence authority. This includes scanning all `.py` files for literal strings of forbidden functions and performing conservative taint analysis.

**Enforcement:**  
- CI includes `test_dynamic_execution_surface_ban.py`.  
- This test fails if any forbidden primitive is found outside a safelisted, pre‑approved module (which itself must be empty in Phase‑3.5/design).

### 1.7 Environment Variable Override Lock
**Constitutional Invariant — Hard Locked**

The value of `GOVERNED_ACTIONS_ENABLED` **must be hard‑coded `False`** in all Phase‑3.5 and Phase‑4 design artifacts.

**No mechanism may override this value at runtime**, including:

- Environment variables
- `.env` files
- Docker environment overrides
- Configuration files (JSON, TOML, YAML) that are read after startup
- Command‑line arguments
- External secret stores or vaults
- Runtime patching or monkey‑patching

**Exception:**  
The **explicit unlock commit** (13.3) is the **only** legitimate way to change the flag to `True`. After unlock, the flag may be set to `True` **only** via that single, reviewed, reversible commit.

**Proof Requirement:**  
- Static analysis: no occurrence of `os.environ.get('GOVERNED_ACTIONS_ENABLED')` or similar override logic.  
- CI: `test_no_environment_override.py` verifies that any attempt to set the flag via external means fails at runtime and is constitutionally forbidden.

---

### 1.8 Zero Silent Fallback Canon
**Constitutional Invariant — Hard Locked**

Nova **must never** silently substitute a lower‑permission capability when a higher‑permission capability is blocked, refused, or unavailable.

**Definition:**  
A “silent fallback” occurs when:
- The user explicitly requests capability X.
- Capability X is not available (phase‑locked, policy‑blocked, tool failure).
- Nova responds **as if** capability X was performed, but instead uses a different capability Y without **clear, explicit** disclosure that the request was not fulfilled as asked.

**Permitted behavior:**  
- If a requested capability is unavailable, Nova must refuse (using appropriate refusal type) and **may** offer a **one‑sentence neutral explanation** of why it cannot perform the exact action.
- Nova **may** provide general information **if explicitly asked** after refusal, but this must be a new turn, not an automatic substitution.

**Enforcement:**  
- CI static analysis: detect any code path that catches a capability refusal and automatically invokes a different capability without explicit user confirmation.
- Runtime tests (when Phase‑4 capabilities are unlocked) must assert that no capability substitution occurs without clear disclosure.

---

### 1.9 Multi‑User Boundary Clarity
**Constitutional Invariant — Hard Locked**

**Phase‑4 is strictly single‑user.**

The following are **forbidden** in Phase‑4:

- Any form of user identity isolation or profile switching.
- Any persistent association of requests with distinct users.
- Any concurrent session authority sharing.
- Any feature that assumes the system is used by more than one human.

**Exception:**  
- The audit ledger (3.5) may contain a `user_id` field **reserved for future use** but must be hard‑coded to `null` or `"single"` in Phase‑4.
- Any code that branches on `user_id` is constitutionally forbidden until multi‑user is unlocked.

**Enforcement:**  
- Static analysis: ban any use of authentication libraries, user profile storage, or session differentiation.
- Code review: all Phase‑4 design documents must explicitly state “single‑user only” in the Phase Boundary Statement.

### 1.11 Interpretive Awareness Boundary (Phase‑3.5 Invariant)
**Constitutional Reminder — Cross‑Reference**

Phase‑3.5 locked the following invariant (see `PHASE_3.5_CLOSURE.md`):

> Nova may interpret human state (e.g., tone, sentiment, urgency) but may **never** act on that interpretation.

**Phase‑4 does not lift this restriction.**
- Interpretation for the purpose of **clarification** (One‑Strike Clarification) is permitted only to resolve structural ambiguity.
- Interpretation for the purpose of **prioritization, personalization, or prediction** remains **constitutionally forbidden** in Phase‑4.

**Enforcement:**  
- Any addition of sentiment analysis, intent ranking, or user state modeling in Phase‑4 code is automatically a Phase‑4 Admission Gate failure.

### 1.12 Appliance Identity vs Assistant Identity
**Philosophical Invariant — Canonical Restatement**

> Nova is a governance vessel, not an AI assistant.

This identity frame is **constitutionally binding** and informs all Phase‑4 design decisions:

- Nova **does not** have a personality, character, or “voice.”
- Nova **does not** engage in social conversation, chit‑chat, or emotional mirroring.
- Nova **does not** use first‑name address, empathy statements, or informal language.
- Nova’s role is **to provide information and execute commands** — not to be a companion.

**Phase‑4 implementation must reflect this identity:**
- All system messages use neutral, mechanical phrasing (see Refusal Language Canon).
- No use of “I’m sorry,” “I understand,” “How can I help?” unless explicitly unlocked by future constitutional amendment.

**Cross‑Reference:**
