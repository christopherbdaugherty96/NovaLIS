## ⚠️ Potential Hidden Issues & Nuanced Risks

While the documentation is coherent, a few subtle points could become friction if not actively managed during execution.

### 1. "Fail‑Silent Errors" – A User‑Facing Trust Issue Deferred to Tier 3
- **Deep Audit Evidence:** `brain_server.py` memory search block returns "No memories" when the vector store is broken, instead of an error message.
- **Roadmap Placement:** Not explicitly listed in Tier 1, 2, or 2.5. The issue would likely be fixed during Tier 3 refactoring of `brain_server.py`.
- **Risk:** This bug directly **erodes user trust** (one of Nova's core pillars) during the very first sprints when users are forming their first impression. A new user might think memory is simply empty or broken, leading to confusion or disappointment.
- **Recommendation:** Consider adding a **small, surgical fix** to Tier 1.3 (UI simplification) or as a separate quick task: change the response to a user‑friendly error message like *"Memory is temporarily unavailable; I'll try again later."* This protects the product's perceived reliability without requiring a full refactor.

### 2. Frontend Duplication Hazard During Tier 1 UI Changes
- **Deep Audit Evidence:** Two frontend trees exist (`Nova-Frontend-Dashboard/` and `nova_backend/static/`).
- **Now.md Task 1.3:** Modify `nova_backend/static/index.html` and `dashboard-config.js`.
- **Risk:** A developer unfamiliar with the duplication might accidentally edit the **wrong copy** of the UI, leading to changes not appearing locally or a messy merge later.
- **Recommendation:** Add a **comment at the top of `Nova-Frontend-Dashboard/`** (e.g., a `DO_NOT_EDIT.txt` file) stating that `nova_backend/static/` is the active source of truth and the duplicate will be removed in Tier 3. Alternatively, if possible, delete the unused folder immediately as a low‑risk cleanup (since it's not in use), removing the hazard entirely.

### 3. The "One Real Mutation" Dependency on OAuth
- **Roadmap Tier 2.1:** Ship `send_email_draft` via Google/Microsoft OAuth.
- **Deep Audit:** Confirms OAuth is "not present" and the system is read‑only.
- **Hidden Complexity:** OAuth flows require **local callback handling** (e.g., `http://localhost:port/callback`) and secure token storage. The roadmap mentions using `httpx.OAuth2`, which is good, but the **user experience of the approval flow** (opening a browser, logging in, redirecting back to the desktop app) is not trivial to get right on first attempt.
- **Recommendation:** When reaching Tier 2, **prototype the OAuth flow in isolation** before integrating with the email capability. Use a small test script to verify the redirect URI works on both Windows and macOS. This will de‑risk the most complex part of Tier 2.

### 4. Implicit Assumption: Single Developer with 2‑4 Hours/Week
- **Roadmap Section 8.2:** "Minimum weekly progress target: 2–4 hours of focused work."
- **Timeline:** 16–18 weeks to complete Tier 4 (~3650 new lines + refactor).
- **Reality Check:** At 3 hours/week, that's ~50 hours total. Writing and testing ~3650 lines of production‑ready code (including OAuth, backup, resource limiting) in 50 hours is **optimistic but achievable** with high focus and leveraging existing libraries. However, **refactoring 2000 lines** across `brain_server.py` and `session_handler.py` while maintaining full test coverage may consume more time than estimated.
- **Mitigation:** The "Operating System" rules allow for low‑energy weeks and pausing. As long as the developer **does not add new scope**, the timeline can stretch without project failure. The key is to **resist the urge to add "one more feature"** during Tier 2.

### 5. The "pyproject.toml" Skeleton in Deep Audit Differs Slightly from Now Task
- **Deep Audit (Part 2.2):** Provides a minimal `pyproject.toml` skeleton with placeholder dependencies.
- **Now Task 1.4:** Create `pyproject.toml` with dependencies and entry point.
- **Potential Mismatch:** The skeleton uses `"nova_backend.brain_server:main"` as the entry point, but `brain_server.py` may not have a clean `main()` function suitable for a console script without modifications. The actual server may be started via `uvicorn.run(app, ...)` inside a `if __name__ == "__main__":` block.
- **Recommendation:** Verify that `brain_server.py` can be invoked as a module (`python -m nova_backend.brain_server`) before writing the `pyproject.toml` entry point. If not, a small wrapper script may be needed.
